# SkillOS 项目思路（整理版）

> 基于 Skill Distiller 原始设计（`PROJECT_DESIGN.md`）与 SkillOS 当前实现状态整理。
> **最后更新**：2026-06-15

---

## 1. 一句话

**SkillOS 不是又一个 Agent 平台——它是给 Agent 造子弹的兵工厂。**

用户用日常语言描述「怎么做一件事」，系统通过对话（或贴链接/丢文件）帮他产出 **可安装、可执行、可带走** 的 `SKILL.md`，直接用于 Cursor、Claude Code、Trae Solo 等工具。

---

## 2. 北极星：唯一核心旅程

所有功能都应服务这一条主路径。其它模块是「让子弹更准、更耐用」，不是第二产品。

```
┌─────────────────────────────────────────────────────────────────┐
│  用户（不懂什么是 Skill）                                         │
│    在 IM / Web / MCP 里用自然语言描述工作流程                      │
└────────────────────────────┬────────────────────────────────────┘
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│  萃取智能体 SkillExtractionAgent                                 │
│    苏格拉底追问 → 场景推演 → 渐进草稿 → 确认 → 生成               │
│    输出：标准 SKILL.md（Instructions + S_route + trigger + params）│
└────────────────────────────┬────────────────────────────────────┘
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│  落盘 + 可移植                                                    │
│    skills/<名称>/SKILL.md  +  YAML description / portable_slug   │
│    导出 → 复制到 ~/.cursor/skills/ 等 → 现场可执行                │
└────────────────────────────┬────────────────────────────────────┘
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│  （增强层，非阻塞主路径）                                           │
│    认识论验证 · 执行轨迹 · 自进化 · 知识扩散 · 变体 · MetaSkill    │
└─────────────────────────────────────────────────────────────────┘
```

**成功标准（产品层）**：

| 指标 | 含义 |
|------|------|
| 10 分钟对话 | 零技能概念用户能产出可用 SKILL.md |
| 安装即用 | 导出文件在 Cursor / Claude Code / Trae 加载后能按步骤执行 |
| 不编造 | 对话未确认的内容标 `[待确认]`，不假装完整 |
| 可带走 | 输出是 Markdown 文件，不绑定 SkillOS 运行时 |

## 3. 我们在解决什么问题（三层问题）

市面 Agent 平台大多回答：**「怎么执行 Skill？」**

SkillOS 回答更上游的三问：

| # | 问题 | SkillOS 手段 |
|---|------|-------------|
| 1 | **Skill 从哪来？** | 对话萃取、URL/文件 7 步学习管线 |
| 2 | **Skill 对不对？** | 认识论分层（Evidence → Knowledge）+ 证伪 + 用户确认 |
| 3 | **Skill 能否越用越好？** | 轨迹评分 → 进化 → SkillOpt 有界编辑 → 跨 Skill 扩散 |

**优先级**：先彻底做好 **① 从哪来 + 可安装输出**，再叠 **② 验证**，最后 **③ 进化**。不能反过来——没有稳定产出，验证和进化是空转。

---

## 4. 设计哲学（精简版）

来自 Skill Distiller 原始设计，作为不可妥协的原则：

### 4.1 卷 Harness，不卷模型

编排、验证、结构化、进化在 **Harness 层** 完成。DeepSeek Flash + 好流程 > 裸 GPT-4 一次性生成。

### 4.2 Skill 即文档

- 唯一交付物是 **Markdown + YAML frontmatter**
- 不依赖 SkillOS 专有运行时才能执行
- Git 可管、人可读、任何 LLM 可加载

### 4.3 咨询式交互，非表单式

- 场景推演 + 可点选项，不是「请填写步骤 1/2/3」
- 双输出：每轮 **一个问题** + **后台草稿增量更新**（只增不改）

### 4.4 知识的可验证性 > 数量

经验（Experience）≠ 知识（Knowledge）。未验证内容不能当真理传播到其它 Skill。

### 4.5 结构从使用中涌现

分类、聚类、DNA 从实际 Skill 与图谱链接中长出，而非预设 taxonomy。

## 5. 架构：四层分工

```
┌──────────────── 交互层 ────────────────┐
│ Web UI · 飞书/微信(Hermes) · MCP · API  │
│ 模式：Create（萃取）· Chat · MetaSkill   │
└──────────────────┬─────────────────────┘
                   ▼
┌──────────────── 智能体层 ──────────────┐
│ SkillExtractionAgent  ← 核心，Create 主路径 │
│ Dispatcher            ← Chat/Agent 路由     │
│ MetaSkill Executor    ← 多 Skill 编排       │
│ 系统 Skill（参考，非主路径执行）              │
│   · skill-creator → 规范与模板               │
│   · brainstorming → Agent 模式优化           │
└──────────────────┬─────────────────────┘
                   ▼
┌──────────────── 知识工程层 ────────────┐
│ 认识论 · 知识图谱 · Pattern Miner/DNA   │
│ 变体 · 进化引擎 · SkillOpt · 学习理论   │
└──────────────────┬─────────────────────┘
                   ▼
┌──────────────── 数据层 ────────────────┐
│ skills/*/SKILL.md · versions/ · traces   │
│ knowledge/ · tenants/ · conversations    │
└──────────────────────────────────────────┘
```

**关键澄清**：

- **Create 模式主路径** = `SkillExtractionAgent`，不是 Dispatcher 调 Skill。
- **brainstorming / skill-creator** 是方法论与规范参考；应在萃取的「提问 / 生成 / 验收」阶段**注入规则**，而非替代萃取智能体。
- **Agent 模式** = 已有 Skill 的日常使用 + 系统优化（brainstorming），与 Create **分路**。

## 7. 产品边界（刻意不做的事）

| 不做 | 原因 |
|------|------|
| 做通用 Agent 平台与 Dify/Coze 正面竞争 | 我们产出 **文件**，不是锁定在平台里 |
| 让用户先学「什么是 Skill」 | 用户只描述流程，系统负责文档化 |
| 萃取时强行跑满 7 步认识论/进化 | 先交付 SKILL.md，验证/进化异步或可选 |
| 一个模式包打天下 | Create / Chat / Meta 分路清晰 |

## 8. 路线图（建议执行顺序）

### Phase A — 核心闭环（当前焦点）

> **让不懂 Skill 的人，对话 → 拿到能装的 SKILL.md**

- [x] 多轮萃取状态持久（SessionManager 单例）
- [x] 苏格拉底 + `[选项]` 按钮
- [x] 确认生成（「可以了」）不误重启
- [x] portable_skill：description、slug、安装路径、导出
- [x] 恢复系统 Skill 文件 + Create/Agent dispatch 分路
- [x] 生成后一键 Zip（含 frontmatter 完整 SKILL.md）
- [x] 端到端验收脚本纳入 CI（`tests/test_phase_a.py`）

### Phase B — 质量与信任

- 认识论：生成后待确认声明 UI 流程顺畅
- skill-creator 规范：生成前 checklist + 生成后 DNA 合规自动修
- `[选项A：…]` 等格式兼容

### Phase C — 增长与企业

- 通道：飞书/微信/Hermes 与 Web 体验一致
- Org 审批、变体、SkillOpt 导出
- 知识扩散、进化（用着用着变好）

## 9. 与竞品的定位（一句话）

| 它们 | 我们 |
|------|------|
| LangChain / CrewAI：用**代码**定义能力 | 用**对话**定义能力 |
| Dify / Coze：能力锁在**平台** | 输出**文件**，可带走 |
| SkillOpt：只优化已有文本 | **创建 → 验证 → 优化 → 关联** 全生命周期 |
| 通用 RAG：存文本块 | **四级认识论** + 证伪 + 跨 Skill 学习 |

**SkillOS 的独特位**：全球少有的 **Skill 工厂**——不只执行 Skill，还负责 **生产、质检、迭代、关联**。

## 附录：名词对照

| 名词 | 含义 |
|------|------|
| **Skill** | 一份 SKILL.md：教 Agent「何时触发、按什么步骤做」 |
| **萃取** | Create 模式下多轮对话生成 Skill 的过程 |
| **Harness** | 编排层：prompt、状态机、验证、进化（非模型权重） |
| **Skill DNA** | 从大量 Skill 提炼的「好 Skill 长什么样」 |
| **MetaSkill** | 多个 Skill 的 DAG 编排，本身也是 Markdown |
| **portable_slug** | 英文 kebab-case 名，供 Cursor/Claude 安装路径使用 |
