# SkillOS 产品哲学

> 一份写给未来的自己、团队、和任何想理解这个产品"为什么是这样"的人的文档。
> 不是设计参考手册（那是 DESIGN.md），而是设计思路的叙事。

---

## 第一章：为什么做 SkillOS

2025年底，Anthropic 发布了 AgentSkills.io 开放标准——用 Markdown 文件定义 AI 助手的能力。到2026年中，超过 85,000 个公开技能出现，30 多个平台宣布兼容，Linux Foundation 在讨论把它纳为正式标准。一个新的软件供应链正在形成：skill 文档就是 AI 的可执行代码。

但所有人都只关注一个问题：怎么执行 skill。Dify 有可视化工作流，Coze 有拖拽编排器，LangChain 有代码级控制。整个行业在"执行层"卷得不可开交。

三个更根本的问题没人管。

第一个问题：skill 从哪来。目前所有 skill 都是手工写的——一个人坐下来，回忆自己的工作流程，把它敲成 Markdown。但大多数人的专业知识是隐性的，他们知道怎么做却说不清楚。更别说从网页、文档、对话记录中自动提取——那需要真正的理解，不是复制粘贴。

第二个问题：skill 对不对。任何人可以写一个 skill 声称它"好"。AI 从一篇博客提取的方法论——它是真知识还是幻觉？是一条经过验证的流程还是某个人的主观偏好？没有人验证。整个生态建立在"相信你看到的"这个脆弱的假设上。

第三个问题：skill 能不能越用越好。软件代码有测试、有 CI、有监控，代码质量随迭代提升。但 skill 是一次性的——写完就放在那，永远不会自己变好。你用了几十次发现了问题，它还是老样子。甚至，当你在一个 skill 中找到改进，其他相关的 skill 完全不知道。

SkillOS 就是回答这三个问题的。我们不是一个"更好的 AI 平台"，我们是给 AI 造子弹的兵工厂。

## 第二章：经验 ≠ 知识——认识论引擎的哲学根基

这是整个产品最根本的设计洞察，也是最容易被低估的部分。外面的人看到"认识论"三个字，大概率会觉得这是过度工程——"一个 skill 管理系统搞什么哲学？"但我们坚持认为，这是 SkillOS 和全世界所有其他技能系统最本质的区别。

让我用一个具体的例子解释。

假设你让 AI 从一篇技术博客提取方法论。那篇博客说："代码审查应该先看测试，再看逻辑，最后检查代码风格。"AI 把这条规则放进了一个 Code Review skill 中。如果你没有验证层，这条规则就直接成为了 skill 的一部分——"skill 说的"就是对的。

但这条规则真的正确吗？它是作者的实践总结还是个人偏好？有没有反例？有没有场景下它不适用？有没有其他独立来源也这么说？如果我们用 AI 去反驳它，它能活下来吗？

SkillOS 引入了认识论层级来系统化地回答这些问题。我们借鉴了四位哲学家的工作，把他们从抽象概念变成了可执行代码。Plato 告诉我们，真信念需要 justification——你不能只是因为有人说就对。Popper 告诉我们，真理不是靠证实，而是靠证伪——一个理论只有在被"攻击"之后还站得住，才值得相信。Kant 提醒我们，我们看到的（现象）不等于真实的（物自身）——AI 提取的内容可能只是表象。Polanyi 指出，人类知道的多于能说出来的——默会知识无法从文本中简单提取。

把这些哲学概念工程化，我们定义了五个认识论等级。最低级是 Evidence（证据），就是从网页、对话、文件中抓取的原始碎片。往上是 Experience（经验），经过基本的分类和关联，但仍然是"某处说的某句话"。再往上是 Knowledge（知识），必须同时满足四个条件才能到达这个级别：至少两个独立来源交叉验证、无矛盾（不和已有知识冲突）、证伪存活（AI 主动反驳失败）、时间稳定（过了一段时间还是对的）。Knowledge 上面是 Preference（偏好）——主观选择，如"用中文还是英文"。错误（Error）和被取代的旧知识（Superseded）也有各自的状态——它们不是被删除，而是被保留下来供审计和回溯。

代码实现了这套验证管线。`record_claim()` 记录每条声明，`promote_to_knowledge()` 检查四个条件，`_falsify_claim()` 让 AI 扮演魔鬼代言人。每一条进入 skill 的"硬规则"——那些 if-then、must-do 级别的步骤——都必须经历过这套流程。而"参考信息"、附录内容可以容错更多，标注为待验证即可。

全球没有任何其他技能系统有这一层。这不是因为我们比别人聪明——是因为我们问了一个别人没问的问题：你怎么知道你的 skill 是对的？

## 第三章：苏格拉底对话——为什么不用表单

很多产品经理看到我们的萃取流程，第一反应是"为什么不做一个表单？让用户填步骤1、步骤2、参数、触发条件，多清晰。"

表单的前提是，你已经很清楚你的工作流程是什么。但实际情况是，多数人最擅长的事情他们是"说不清楚"的。一个做了十年合同审核的律师，她看到一份合同时，直觉就知道第七条有问题。但你问她"你是怎么判断的"，她可能需要想很久才能说出一二三四。这就是 Polanyi 说的"默会知识"——我们知道的比我们能说的多。

直接说"请描述你的步骤"——大概率得到的是一堆模糊的、不完整的、遗漏了关键判断点的文字。苏格拉底的方法不是直接问答案，而是通过提问帮助你自己发现你已经知道但没表达出来的知识。

我们的对话不是模板式的问卷。它更像一个有经验的同事。它不问"这个流程有哪些步骤"，而是给一个场景："假设对方发来采购合同，第7条把北京仲裁改成上海仲裁——按你的想法，skill 第一步该做什么？这属于什么风险等级？"然后给出几个选项让你选。每轮对话，AI 做两件事：第一，问一个场景推演式的问题；第二，在后台逐步完善技能草稿。你能看到问题、能做出选择、能补充细节——但你看不到 AI 在后台已经把草稿从三句话扩到了一页。

这个双输出模式的核心承诺是"只增不改"——已有的正确内容不动，新信息往上加，不确定的标 `[待补充]`。"待补充"是一个诚实的标记——它告诉用户"我知道这里还缺东西，你需要补上"，而不是让 AI 假装它都懂了。

场景推演还有一个关键的价值：它把抽象的"最佳实践"变成了具体的"我遇到这种情况会怎么做"。这不是 AI 在教你做事——是 AI 帮你把你脑子里那个"懂但说不清"的东西挖出来。

## 第四章：七步学习——AI 怎么"真正理解"一个网页

一次 LLM 调用可以直接说"总结这篇文章并生成技能文档"。很多工具就是这么做的。但这样做的问题是什么？

你没有真正"理解"——你只是在复制。

人类学习新知识不是这样的。你不会打开一篇物理学论文，扫一眼，然后合上书说"好了，我懂了"。你会先判断这文章值不值得读（初识），然后问"为什么"而不是"是什么"（理解），接着把它拆成可操作的步骤（拆解），用自己的话重新组织（重构），测试你的理解对不对（验证），把它和已有的知识体系关联起来（内化），最后形成可以随时调用的能力（沉淀）。

SkillOS 的七步 URL 学习管线模拟的正是这个过程。每一步有不同的 token 预算——不是平均分配。初识只给 200 token，因为它只需要一个快速的"值得/不值得"判断。重构给 2000 token，因为这是最需要深度思考的一步——把作者的话变成执行者能用的指令。这个逐级递增的预算设计不是拍脑袋的结果，它来自 Bloom 的教育目标分类学（记忆→理解→应用→分析→评价→创造）——每一步比前一步需要的认知深度更高。

除了 Bloom，另两个灵感来源是 Feynman 技巧和刻意练习。Feynman 说"说不清楚的地方就是理解不够"——所以我们有重构步骤，系统性地检查"这个地方你能不能用简单的话说清楚"。刻意练习的核心是"验证→反馈→调整→重复"——所以我们有验证步骤，让 AI 对生成的 draft 做对抗性攻击：边界的 case 在哪？步骤顺序有没有问题？有没有歧义？有没有缺失的依赖？

第八步是"扩散"——这一步画龙点睛。你不能学了一个新东西就完事了。你得问：这个新学到的能不能改善我已经会的那些？在 SkillOS 中，每当你学会了一个新 skill，系统会自动检查你能不能帮到已有的 5 个相关 skill。这是在模拟人类学习的"迁移"——把在一个领域学到的洞察应用到另一个领域。

## 第五章：技能为什么需要进化

软件代码有版本管理、有 CI/CD、有 bug 修复流程。skill 为什么不能？

一个 skill 发布了 v1.0。用户用了 50 次。有些成功，有些失败。每次失败都有一个 trace 记录发生了什么、在哪个步骤出错的、用户的反馈是什么。这些数据散落在日志里。一个人类开发者也许永远不会去翻这些日志。但 AI 可以。

SkillOS 记录每一次执行的 trace。当某个 skill 连续失败 3 次，系统自动触发进化流程：诊断失败根因——是不是某个步骤在新场景下不适用？是不是某个参数的建议值太极端？是不是两个步骤的顺序需要调整？然后 LLM 生成改进版，验证门把关（改进版必须比原版好），通过后保存新版本。

但不是所有 skill 都用同一种进化方式。新 skill 可能积累了大量失败轨迹，需要批量诊断——Trace2Skill 擅长做这个。不稳定 skill，分数忽高忽低，可能是某个子步骤在边界条件下变差——EvoSkill 生成多个候选让它们在真实场景中竞争，强者保留。成熟的稳定 skill 已经有了良好的基线，不需要大改——SkillOpt 做外科手术式的精准小改，每次只调整一小段，编辑预算递减（前四次多一点，后面越来越少）。

系统自动选择使用哪种策略。7 条路由规则基于：执行次数、分数方差、成熟天数、遗忘状态。

还有一个更深的洞察：技能之间的知识不是孤岛。当你改进了"客户退款处理"技能的一个步骤，系统会自动检查"这个改进能不能帮到订单查询？能不能帮到支付对账？"如果能，自动应用改进。这就是我们说的"知识扩散"——群体学习。一个 skill 从失败中学到的教训，可以立即赋能所有相关 skill。这是人类团队做不到的事情——没有 PM 有精力去"每改一个流程就检查所有相关流程"。

这种"自进化+知识扩散"的设计，混合了三种不一样的哲学。Trace2Skill 是归纳法的极致——并行分析大量轨迹，层次化合并，多人提到的才进报告。EvoSkill 是达尔文自然选择——变异产生多样性，环境选择最优。SkillOpt 是机器学习的方法论——学习率约束、验证 gate、负反馈 buffer、动量，本质上就是把 SGD 的思想用在 skill 优化上。

## 第六章：多态——同一个问题可以有不同解法

做一个"客服回复"的 skill——应该怎么做？

客服主管会说：关键词匹配 → 语义理解，回复风格要正式。NLP 工程师会说：关键词匹配 → 意图识别 + 实体提取，回复风格要技术性。一线客服会说：关键词匹配 → 先看最近 3 条聊天记录，回复风格要口语化。

传统的做法是统一——选一套"最好的"，或者产品经理拍板。但谁说了算？凭什么客服主管的标准比一线客服的"更好"？也许在不同场景下、不同客户面前、不同紧急程度上，不同版本各有所长。

SkillOS 的做法是求同存异。我们借鉴了三个看似不相关的思想。Plato 的理型论：Archetype 是理想 Form，Variant 是具体实现——我们先把"客服回复应该包含哪些核心步骤"定下来（这是 Archetype），然后每个人可以有自己的实现（这是 Variant）。Wittgenstein 的家族相似性：技能之间不需要共享一个"本质"，它们只需要像家族成员一样有重叠的相似性——"客服回复"和"技术工单处理"不是"同一个技能的不同版本"，它们是有家族相似性的不同技能。Java 的 Interface/Concrete/@Override 工程化：我们定义了 Archetype 应该包含哪些"方法"（步骤），每个 Variant 可以 @Override 其中某一步，标注为什么不同。

最终哪个版本更好？让数据说话。系统追踪每个 variant 的执行成功率。"主管版"在正式沟通场景下成功率 92%，"一线版"在快速响应场景下成功率 87%。用户不需要选——系统自动根据上下文选择最合适的 variant。

## 第七章：前端的渐进式现代化

SkillOS 的前端是一个诚实的、正在转型中的代码库。它不是完美的，但它知道自己的问题在哪。

24 个 JS 文件、2 个 HTML 文件、1 个 CSS 文件——全部加载在一个 65KB 的 `index.html` 中。没有 ES modules，没有打包工具，没有构建步骤。所有的 JavaScript 函数共享一个全局命名空间。这是它出生的方式——快速迭代、零构建、打开浏览器就能用。当项目从一个人几周的工作变成一个需要长期维护的产品时，这种架构的短板开始显现。

我们选择了一条务实的路径：不是重写，而是渐进式迁移。Alpine.js 被引入作为"桥接层"——既保留了现有代码的工作方式（`_currentSkill`、`_mode` 等全局变量），又通过 `window.__alpineReady` 和 `Object.defineProperty` 透明地把它们映射到 Alpine store。这意味着新代码可以写 `Alpine.store('nav').currentSkill`，老代码可以继续用 `_currentSkill`——它们读的是同一个值。

CSS 是一个亮眼的设计。完整的 CSS 变量体系定义了从颜色到间距到字号到圆角的所有 token。暗色主题、220ms 过渡动画、骨架屏、禁用态、loading 旋转——这些都是后来叠加上去的，但它们都有变量根基。2026年6月的设计审计发现了 56 处硬编码颜色值——铜色 accent 的 RGB 被直接写在规则中 40 多次。修复很简单：添加 `--accent-rgb` 变量，全局替换。

安全性审计发现了另一个问题：15 处 onclick 字符串拼接直接将技能名注入 HTML 属性——`onclick="showDetail('` + name + `')"`。一个命名为 `'); alert(1)` 的技能就可以触发 XSS。修复方案是统一使用 `JSON.stringify()` 做 JavaScript 参数安全化，用 `escHtml()` 做 HTML 内容安全化。

我们还顺手加了键盘快捷键（Ctrl+K 搜索、Ctrl+N 新建、Escape 关闭弹窗）、统一了缓存破坏参数、补了 9 处 aria-label、把底栏从全局显示改成了只对话页面显示。这些不是大工程，但累积起来，前端从一个「能用就行」的 prototype 变成了一个「合格的产品界面」。

## 第八章：API 层的安全与分级认证

在 2026 年 6 月的安全审查中，我们发现了一个尴尬的事实：`knowledge.py` 的 21 个端点全部没有认证。任何人不需要登录就能查询知识图谱数据、血缘记录、摄入队列、公众号监控列表。

这不是疏忽——这是从 prototype 到产品的过程中被遗忘的一步。最初这些端点只是开发者自己调用，安全性不需要。但当有了多租户支持、有了真实用户、有了外部 MCP 客户端接入，情况完全不同了。

我们引入了分级认证体系。公开端点（登录、注册、文档、语音转录）完全不需要认证。GET 类查询端点（技能列表、知识查询、市场目录）使用 optional auth——未登录用户可以访问公共内容，登录用户自动启用租户过滤。POST 类修改端点（创建技能、触发进化、运行消化循环）使用 required auth——必须携带有效 JWT。

同时给 MCP 服务器加了两个防护：`fetch_url` 工具阻止 SSRF——不能访问 localhost、127.0.0.1、192.168/16、10/8 等内部地址。`ingest_file` 工具加了路径穿越保护——只能读取 workspace、home、Downloads 目录下的文件。

这些改动不大——每个就几行代码。但这是产品从"开发者工具"变成"可发布的软件"的关键一步。

## 第九章：质量基础设施——从 0 到 1

一个项目的代码质量，看它有什么"护栏"就知道了。

2026年6月初，SkillOS 的代码质量基础设施是：pytest 测试（605 个，98.3% 通过率），mypy（13 个禁用码，号称 0 error 但实际是关了检查），pre-commit 有配置文件但没安装。没有 CI 本地复现方式。120 个文件在 git 状态下显示为修改——大部分是运行时产生的 lineage 数据和 skill 版本快照，不该被版本控制。

6月底，状态完全不同了。

mypy 从 13 个禁用码减到 2 个。11 个被消灭的禁用码中，有些是死配置（`call-overload` 从来不触发），有些是修完一个错误就能移除一个（`name-defined` 只有 1 个错误的变量名引用），有些是添加了几十个类型注解（`var-annotated` 12 处补丁）。剩下 2 个是真正需要重构的硬骨头——`attr-defined` 的 34 个错误中包括 `SkillDNA.dna` 和 `StepConfidence.step` 这种属性不存在的设计问题，`arg-type` 涉及 OpenAI SDK 的严格类型签名和 `max()` 的重载函数。

Pre-commit 安装并跑通了。919 个文件被自动修复了尾随空格和文件结尾换行符。Git 状态从 120 个脏文件降到了 0。

SkillsBench 基准测试——项目最核心的质量门禁——从 v0.3.3 的 Reference Quick8 三门禁全部失败恢复到了 ALL PASS。Reference 技能获得 +28/+18.6/+22 个百分点的提升。Layer 1 ablation 实验量化了 HERITAGE + pack 路由的贡献：中位 Δ 从 0 提升到 +45。

GitHub remote 配置并推送了——之前 4 条 AI_DEV_LOG 记录都写着"未修改/刻意不做：git push（无 remote）"。

还加了一个 `ci_local.sh`，把 GitHub Actions 的六阶段管道复现到了本地。

最后——也是最有意思的一个变化——我们给项目配置了 Claude Code 自动化。三个 hooks（编辑后自动 ruff、编辑 .env 被阻止、会话结束时 AI_DEV_LOG 提醒）、两个 skills（自动写日志、跑 bench 回归）、两个 agents（安全审查、测试生成）、两个 MCP 服务器（context7 实时文档、GitHub Issue/PR 管理）。这些不是基础设施本身——而是让 AI 协作变得无缝的工具。

## 第十章：下一步

产品化路线图分三层。

底层是稳定。mypy 的 2 个剩余禁用码需要逐步消灭——但这不是简单加注解，而是需要重构类结构（`SkillDNA.dna` 为什么不存在？`StepConfidence.step` 去哪了？）。前端需要模块化——24 个 JS 文件共享全局命名空间迟早会出问题。但模块化不能破坏现有的 Alpine 桥接层——它正在从旧架构向新架构过渡。

中层是验证。认识论引擎需要更多实证数据。Layer 1 ablation 已经证明了 HERITAGE + pack 路由的量化贡献，但费曼+类比技巧的边际效应还没测。论文需要更新 Layer 1 实验节。test feasibility eval 已经修好了（4/4 pass），`--ignore` 标志已移除。但 2 个瞬态失败测试（LLM 超时）需要更系统的处理——目前的 retry 是一个过渡方案。

上层是增长。前端已经做到了"合格的产品界面"，但还有 10 个 UI 缺陷在审计文档中躺着。IM 通道的飞书和微信集成需要真实验证。多租户功能需要更完善的治理面板。市场的订阅、审核、定价流程需要打磨。

但核心的方向不改变：**以 bench 回归与产品化为主**。不新增 Phase，不扩范围，在已有系统上做深做透。
