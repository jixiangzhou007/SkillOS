# 技能名称：测试抽取流程（Test Extraction Flow）- 草稿 v0.4

## 元数据
- **类型**: MetaSkill 流水线
- **版本**: 0.4
- **触发条件**: 用户提交订单时触发
  - [待补充] 触发方式：是监听事件，还是接收API调用？
- **核心目标**: 从用户提交的订单数据中，按照既定规则准确抽取目标信息，并处理边界情况。

## 流水线步骤

### 阶段一：前置条件与输入定义
- **输入**:
  - 用户提交的完整订单数据（结构化为JSON或表单）。
  - [已确认] 包含订单ID和客户ID。
  - [待补充] 是否包含文件（如合同、凭证）？需要用户确认。
- **前置条件**:
  - 订单数据已通过基础校验（如必填字段不为空）。
  - [待补充] 抽取规则（如字段优先级、冲突处理策略）是否已预定义？

### 阶段二：执行步骤
1.  **数据预处理与格式校验**:
    - 检查订单ID、客户ID等关键字段的格式是否符合预期（例如：长度、字符类型）。
    - **边界情况处理**:
      - **若格式校验失败**:
        - [待补充] 用户需选择处理策略：直接报错终止、跳过记录、或暂停等待人工审核。
2.  **规则匹配**: 根据预定义的抽取规则，扫描订单数据中的目标字段。
    - [新增] **参数化规则**：规则不应是硬编码的。用户应通过参数提供“字段映射表”（例如：目标字段“商品名称”对应源字段的“product_name”或“item_desc”）。
3.  **冲突检测**: 检查是否存在名称相似或语义重叠的字段（例如“商品A（主商品）”与“商品A（赠品）”）。
    - **边界情况处理**:
      - **若存在冲突**:
        - 第一步：判断规则库中是否有针对此类冲突的优先级定义。
          - 若已定义（例如“主商品优先于赠品”），则按规则处理，并记录处理日志。
          - 若未定义，则**暂停流程，输出冲突字段详情，并报错等待人工介入**。
        - [新增] 风险等级：字段冲突且无规则定义时，风险等级为 **risk_high**。
      - 若规则未定义，则报错并暂停流程，等待人工定义规则。
    - [待补充] **同物异名处理**：当同一目标存在不同源字段名时，用户需通过参数提供“别名映射表”或选择处理策略（自动归并/报错）。
4.  **数据抽取**: 从订单数据中提取匹配到的目标字段值。
5.  **格式转换（可选）**: [待补充] 根据下游系统要求，将抽取结果转换为指定格式（如纯文本、JSON）。
