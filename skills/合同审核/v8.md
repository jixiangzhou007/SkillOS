# 合同审核

## 核心问题
审核对方发来的销售合同（软件集成类项目），识别价格、质保、免责、知识产权、违约金等关键条款的风险，并给出修改建议。

## When to use
- keywords: 合同审核, 销售合同, 合同风险, 审核条款, 合同审查, 合同修改, 合同谈判
- context: 用户收到对方发来的销售合同（软件集成类项目），需要审核关键条款风险并给出修改建议
- excludes: 用户询问的是采购合同（非销售）、用户需要起草合同（非审核）、用户需要法律诉讼建议

## Instructions
Follow these steps in order. Ask the user if anything is marked [待确认].

1.  **接收合同文件并确认基本信息**
    -   确认用户上传了合同文件（PDF/Word/图片）。
    -   询问用户：合同对方全称、合同总金额、项目大致范围（软件/硬件/服务）。
    -   若用户未提供，标记为 `[待确认]` 并继续后续步骤。

2.  **审查合同标的与价格条款**
    -   **检查标的描述**：合同中的产品/服务描述是否清晰（含规格、版本、功能清单、交付物明细）。
        -   若模糊 → 标记为 **高风险**，建议要求对方以附件形式提供《产品规格与功能清单》。
    -   **检查价格与税务**：总价是否含税？是否注明税率和发票类型（专票/普票）？
        -   若未注明 → 标记为 **中高风险**，建议明确约定“本合同总价为含13%增值税专用发票的价格”（或根据实际情况调整）。
    -   **检查付款方式**：是否为100%预付款？
        -   若是 → 标记为 **高风险**，建议修改为分阶段付款（如30%预付+60%验收+10%质保），或要求履约保函。
        -   若为分期付款 → 检查各阶段比例是否合理（预付比例不宜过高）。

3.  **审查质保条款**
    -   **质保期限**：质保期多长？（软件集成类通常1-3年）
        -   若质保期过长（>3年）或未明确 → 标记为 **中风险**，建议明确质保起止时间和期限。
    -   **质保范围**：是否覆盖软件升级、bug修复、硬件更换？
        -   若范围模糊 → 标记为 **中风险**，建议明确质保服务内容清单。
    -   **响应时间**：是否有明确的故障响应和修复时间（SLA）？
        -   若无 → 标记为 **中风险**，建议增加SLA条款（如：2小时响应，24小时修复）。

4.  **审查免责条款**
    -   **免责范围**：免责条款是否过于宽泛（如“任何不可抗力均免责”）？
        -   若是 → 标记为 **高风险**，建议限定不可抗力的具体范围（如：自然灾害、战争、政府行为），排除可预见的商业风险。
    -   **免责上限**：免责金额是否远低于合同总价？
        -   若免责上限过低（如仅为合同金额的10%）→ 标记为 **高风险**，建议争取与合同总价挂钩（如100%）。
    -   **间接损失免责**：是否排除了间接损失赔偿？
        -   若完全排除 → 标记为 **中风险**，建议争取对直接损失（如数据丢失、系统瘫痪）保留索赔权。

5.  **审查知识产权归属条款**
    -   **背景知识产权**：我方在项目中使用的自有技术/软件，知识产权是否明确归我方所有？
        -   若未明确 → 标记为 **高风险**，建议增加“双方各自保留背景知识产权”条款。
    -   **前景知识产权**：项目开发过程中产生的新成果（定制代码、方案设计），知识产权归谁？
        -   若归对方所有且未给予我方使用许可 → 标记为 **高风险**，建议争取共同所有或我方保留使用权。
    -   **侵权保证**：对方是否保证其提供的产品/软件不侵犯第三方知识产权？
        -   若无 → 标记为 **高风险**，建议增加侵权保证和赔偿条款。

6.  **审查违约金条款**
    -   **对等性**：双方的违约金条款是否对等？（如：我方逾期交付罚X%，对方逾期付款罚Y%）
        -   若不对等（我方罚则重，对方轻或无）→ 标记为 **高风险**，建议修改为对等条款。
    -   **违约金上限**：违约金是否有上限？上限是否合理？
        -   若无上限或上限过高（>合同总价30%）→ 标记为 **中风险**，建议设置合理上限（如合同总价的20%）。
    -   **触发条件**：违约金的触发条件是否清晰、可执行？
        -   若模糊（如“严重违约”未定义）→ 标记为 **中风险**，建议明确具体违约情形。

7.  **输出审核报告**
    -   汇总所有风险点，按 **高风险**、**中风险**、**低风险** 分类。
    -   对每个风险点给出：
        -   风险描述（引用原文）
        -   风险等级
        -   修改建议（具体措辞或谈判话术）
        -   边界条件（什么情况下可接受）
    -   提供整体建议：是否建议签署、需要重点谈判的条款、可让步的条款。

## Decision routes
| 用户意图/条件 | 执行动作 | 备注 |
|------------|---------|------|
| 用户上传合同文件 | 执行 S_body 第1步，确认基本信息后依次审查各条款 | 默认入口 |
| 用户询问“这个合同有什么风险” | 直接执行 S_body 第2-6步，输出风险清单 | 跳过基本信息确认 |
| 用户指定审查某个条款（如“看看质保条款”） | 仅执行对应条款的审查步骤（如第3步） | 按需执行 |
| 用户说“这个条款怎么改” | 对指定条款给出具体修改措辞和谈判话术 | 基于已审查结果 |
| 用户说“这个风险可以接受吗” | 根据边界条件判断，给出接受/拒绝建议 | 需结合项目背景 |

## Inputs
- contract_file: file, 必填，合同文件（PDF/Word/图片）
- counterparty_name: string, 可选，合同对方全称
- total_amount: number, 可选，合同总金额（元）
- project_scope: string, 可选，项目范围描述（如“软件+硬件+实施服务”）
- review_focus: string, 可选，指定审查重点（如“只看质保和知识产权”），默认全部审查
- risk_tolerance: string, 可选，风险容忍度（“保守”/“平衡”/“激进”），默认“平衡”
