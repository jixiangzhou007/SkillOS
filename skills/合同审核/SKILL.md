# 合同审核

## 核心问题
快速识别销售合同中的关键风险条款，并给出修改建议。

## S_trigger
- keywords: contract, 合同, 审核, 条款, 风险
- context: 用户上传或提及一个文件，意图是进行合同风险审查，尤其是销售合同。
- excludes: 用户询问法律咨询意见（非合同审核）、用户要求起草新合同（非审核现有合同）、文件名明确为其他类型文档（如“会议纪要”）。

## S_body
Follow these steps in order. Ask the user if anything is marked [待确认].

1.  **接收并识别合同文件**：
    -   接收用户提供的文件。
    -   **如果**文件名包含 “contract”（不区分大小写），则进入合同审核流程。
    -   **否则**，提示用户：“当前文件似乎不是标准销售合同（文件名未包含‘contract’），是否仍要继续审核？”
2.  **执行合同审核**：
    -   读取合同全文。
    -   按以下顺序检查关键条款，并记录发现的问题和修改建议：
        -   **价格条款**：检查金额、支付方式、支付周期、税率是否清晰明确。
        -   **保修条款**：检查保修期限、保修范围、责任限制是否合理。
        -   **责任条款**：检查责任上限、免责范围、赔偿条件是否对己方有利。
        -   **知识产权条款**：检查技术成果、软件、文档的归属和使用权是否清晰。
        -   **违约条款**：检查违约金比例、触发条件、争议解决方式是否公平。
    -   **如果**发现任何条款存在风险或模糊不清，**则**在报告中高亮标记，并提供具体的修改建议。
    -   **如果**所有条款均清晰合理，**则**在报告中注明“未发现明显风险”。
3.  **生成并输出审核报告**：
    -   将审核结果整理成结构化的报告，包含：合同名称、审核日期、风险等级（高/中/低）、各条款审核详情及修改建议。
    -   将报告呈现给用户。

```markdown
| 用户上传文件，文件名包含 “contract” | 执行步骤1（接收）→ 步骤2（审核）→ 步骤3（输出报告） | 标准流程 |
| 用户上传文件，文件名不包含 “contract” | 执行步骤1（接收）→ 提示用户确认 → 若用户确认，继续审核；否则终止 | 异常处理流程 |
| 用户直接要求“审核这份合同” | 执行步骤1（接收）→ 步骤2（审核）→ 步骤3（输出报告） | 忽略文件名检查，直接进入审核 |
| 用户上传的文件内容为空或无法解析（如文件损坏、格式错误） | 执行步骤1（接收）→ 提示用户：“文件内容为空或无法解析，请重新上传有效合同文件。” → 终止流程 | 空输入/文件损坏兜底 |
| 用户上传的文件内容包含大量乱码或非合同文本（如纯图片、二进制数据） | 执行步骤1（接收）→ 提示用户：“文件内容无法识别为合同文本，请上传可编辑的文档格式（如 .docx, .pdf 文本层）。” → 终止流程 | 非文本内容兜底 |
| 用户同时上传多个文件（如批量审核） | 执行步骤1（接收）→ 提示用户：“当前仅支持单份合同审核，请指定一个文件。” → 等待用户选择单一文件后继续 | 并发/多文件兜底 |
| 用户上传文件后，在审核过程中突然中断或超时（如网络断开、长时间无响应） | 执行步骤1（接收）→ 记录当前进度 → 提示用户：“审核过程异常中断，请重新上传文件并重试。” → 终止流程 | 异常中断兜底 |
| 用户上传的文件名包含 “contract” 但文件内容实际为其他类型（如会议纪要、报价单） | 执行步骤1（接收）→ 步骤2（审核）→ 若内容明显非合同，提示用户：“文件内容与合同格式不符，请确认是否为销售合同。” → 等待用户确认后继续或终止 | 内容与文件名不符兜底 |
```

## S_params
- file_path: string, 必填，待审核合同文件的路径或内容。
- review_focus: string, 可选，默认值 “all”，指定审核重点（如 “price”, “warranty”），若未指定则审核全部条款。

## Outputs
- status: enum — 执行结果（success / pending / rejected / escalated）
- message: string — 给用户或下游系统的摘要说明
- audit_notes: string — 关键操作记录（可选）
