# 工单处理

## 核心问题
标准化处理客户通过邮件或Zendesk提交的技术支持工单，确保分类准确、响应及时、升级规范、关单有据。

## When to use
- keywords: 工单, 技术支持, 故障, 咨询, 功能请求, Zendesk, 邮件, 问题, 报错, 无法登录, 求助, 支持
- context: 当客户通过邮件或Zendesk提交技术支持问题，或用户需要处理技术支持工单时使用
- excludes: 非技术支持类工单（如：销售咨询、账单问题），或用户明确要求使用其他流程处理

## Instructions
Follow these steps in order. Ask the user if anything is marked [待确认].

1. **工单接收与初步分类**
   - 解析工单标题和描述，提取关键问题关键词（例如：“无法登录”、“报错”、“License过期”）。
   - 根据关键词进行初步分类，分类包括：
     - `type_incident`：故障类，系统无法正常使用或出现错误（如：无法登录、页面报错、服务中断）。
     - `type_inquiry`：咨询类，客户需要操作指导或了解产品信息（如：如何导出报表、功能说明）。
     - `type_feature`：功能请求类，客户希望增加新功能或优化现有功能（如：希望增加批量导入功能）。
   - 若信息不全（如缺少日志、截图、产品版本号），使用模板邮件向客户索取，模板需包含：产品版本号、日志时间范围、截图要求。
   - 输出：初步分类标签（`type_*`）及风险等级（P1/P2/P3）。

2. **问题诊断与排查**
   - 条件：基于初步分类结果，执行不同的诊断路径。
   - 动作：
     - 如果是 `type_incident`（故障类）：
       - 优先检查系统服务状态、网络连通性及客户本地环境（如：浏览器缓存、插件冲突）。
       - 同时启动30分钟升级倒计时，若超时未找到方案则升级至二线工程师。
     - 如果是 `type_inquiry`（咨询类）：
       - 从知识库搜索匹配的操作指南或常见问题解答，搜索时必须按产品线和版本号过滤。
       - 若匹配度低于80%，转人工客服处理。
     - 如果是 `type_feature`（功能请求类）：
       - 记录需求并转产品团队，不计入故障SLA。
   - 输出：诊断结果或下一步行动项（如：待客户提供信息、已定位原因、等待二线介入）。

3. **升级二线（条件执行）**
   - 条件：当一线工程师无法在规定时间内解决，或问题超出能力范围时触发。
   - 动作：
     - 收集完整对话记录、已尝试的排查步骤及结果、客户环境信息（系统版本、网络拓扑、日志片段）。
     - 将以上信息附在工单备注中，并转派至二线工程师队列。
   - 输出：升级后的工单状态及二线责任人。

4. **工单备注与状态管理**
   - 每次处理工单时，备注必须遵循固定格式：**问题摘要 / 处理步骤 / 结论 / 待办**。
   - 若发现新工单与已有工单重复，将其合并到母单，并保留所有时间线记录。
   - 若客户情绪激动（如：反复催促、使用感叹号、负面评价），先执行安抚话术，暂缓深入技术排查，待客户情绪平复后再继续。
   - 涉及数据安全或合规的问题走专门通道，禁止口头承诺修复时间。
   - 多语言工单优先用客户使用的语言回复。
   - 周末只有值班工程师，P1故障可以电话联系客户。
   - 与研发协作时，在Jira创建bug并关联原Zendesk工单。
   - 输出：格式化的备注内容。

5. **关单确认**
   - 条件：所有排查和处理步骤完成。
   - 动作：
     - 在关单前，必须获得客户确认问题已解决，否则保持工单状态为 `open`。
     - 若为严重故障（`type_incident` 且风险等级为P1），需同步通知客户经理和客户成功团队。
   - 输出：最终工单状态（`closed` 或 `open`）及通知记录。

## Decision routes
| 用户意图/条件 | 执行动作 | 备注 |
|------------|---------|------|
| 客户提交故障类工单（type_incident） | 先查Confluence知识库，30分钟内无方案则升级二线 | 同时启动30分钟倒计时 |
| 客户提交咨询类工单（type_inquiry） | 按产品线和版本号过滤知识库，匹配度低于80%转人工客服 | 匹配度阈值80% |
| 客户提交功能请求类工单（type_feature） | 记录需求并转产品团队，不计入故障SLA | 不触发升级流程 |
| 工单信息不全 | 使用模板邮件索取日志、截图、产品版本号 | 模板需包含版本号、日志时间范围、截图要求 |
| 客户情绪激动 | 先执行安抚话术，暂缓深入技术排查 | 待情绪平复后再继续 |
| 发现重复工单 | 合并到母单，保留所有时间线记录 | 母单保留完整历史 |
| 涉及数据安全或合规问题 | 走专门通道，禁止口头承诺修复时间 | 需遵循合规流程 |
| 多语言工单 | 优先使用客户使用的语言回复 | 保持语言一致 |
| 周末P1故障 | 值班工程师可电话联系客户 | 仅限P1级别 |
| 需要研发协作 | 在Jira创建bug，关联原Zendesk工单 | 双向关联 |
| 关单前 | 必须获得客户确认问题已解决 | 否则保持open状态 |
| 严重故障（P1）关单时 | 同步通知客户经理和客户成功团队 | 确保通知到位 |

## Inputs
- ticket_id: string, 格式为 `INC-` 开头（如 `INC-20231001`），工单唯一标识
- ticket_type: enum, 可选值 `type_incident` / `type_inquiry` / `type_feature`，工单分类
- risk_level: enum, 可选值 `P1` / `P2` / `P3`，风险等级（P1: 四小时内首次响应，P2: 当天，P3: 三个工作日内）
- customer_language: string, 默认 `zh-CN`，客户使用的语言
- knowledge_base_filter: object, 包含 `product_line` 和 `version`，知识库搜索过滤条件
- escalation_timeout: int, 默认 `30`，升级倒计时（分钟）
- faq_match_threshold: float, 默认 `0.8`，FAQ匹配度阈值
- jira_link: string, [待确认] Jira缺陷单关联ID
- compliance_channel: string, [待确认] 数据安全/合规问题的专门通道标识
