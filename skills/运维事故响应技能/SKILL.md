# 运维事故响应技能

## 核心问题
标准化处理线上告警事故，确保定级准确、响应及时、根因可追溯、改进可落地。

## When to use
- keywords: 告警, 事故, P0, P1, 线上故障, 系统异常, 紧急响应, 宕机, 报错, 资损, 响应, 修复, 灰度, 复盘
- context: 用户报告线上系统异常、收到监控告警、需要处理运维事故时
- excludes: 用户询问一般性技术问题、非紧急的日常运维操作、计划内的变更发布

## Instructions
Follow these steps in order. Ask the user if anything is marked [待确认].

1. **告警定级（5分钟内完成）**
   - 获取告警信息、服务名、时间窗口
   - 使用快速决策树判断级别：
     - 问题1：是否为核心业务功能（如支付、订单、登录）？ → 是则倾向P0
     - 问题2：错误率/延迟是否在持续恶化（5分钟内上升超过5%或线性增长）？ → 是则倾向P0
     - 问题3：影响范围是否超过1%用户？ → 是则倾向P0
     - 以上均否 → 定为P1
   - 若不确定，宁高勿低：先按P0拉群，确认后降级
   - 定级结果记录：时间、级别、决策依据

2. **响应启动**
   - 若为P0：立即拉群通知全员，要求5分钟内响应，15分钟内定位根因
   - 若为P1：由值班同学主导处理，必要时升级
   - 设置30分钟计时器：若30分钟无进展（未定位根因或未开始修复），自动升级处理

3. **根因定位（15分钟内完成）**
   - 查看日志：按traceId、错误码、时间窗口聚合搜索
   - 查看监控指标：交叉对比CPU、内存、慢查询、上游调用量
   - 确定影响范围：受损功能、影响用户数、错误率、延迟数据
   - 记录定位结果：根因、影响范围、证据链

4. **方案修复**
   - 制定修复方案（回滚、热修复、扩容等）
   - 灰度发布：先1%流量观察5分钟，确认无新告警或指标恶化
   - 逐步放量：10% → 50% → 全量，每阶段观察3-5分钟
   - 回滚条件：灰度期间出现新告警或指标恶化，立即回滚
   - 记录修复动作：方案、灰度过程、回滚记录（如有）

5. **验证确认**
   - 确认告警恢复、错误率归零、延迟恢复正常
   - 观察5-10分钟确认无反复
   - 记录恢复时间点

6. **复盘总结**
   - 使用复盘报告模板撰写报告：
     - 事故时间线（告警触发→定级→定位→修复→恢复）
     - 根因分析
     - 影响范围（功能、用户数、时长、资损[待确认]）
     - 修复动作
     - 改进措施（监控完善、代码修复、流程优化等）
     - 责任人
   - 沉淀改进措施到知识库或工单系统[待确认]

## Decision routes
| 用户意图/条件 | 执行动作 | 备注 |
|------------|---------|------|
| 用户说「告警了」「出事故了」「P0/P1」 | 启动定级流程，获取告警信息 | 先问清楚告警内容、服务名、时间窗口 |
| 用户说「定位到了」「找到根因了」 | 记录根因和影响范围，进入方案修复 | 确认是否有修复方案 |
| 用户说「修复了」「灰度通过了」 | 进入验证确认步骤 | 确认告警是否恢复 |
| 用户说「恢复了」「告警消了」 | 进入复盘总结，要求用户提供复盘信息 | 按模板生成复盘报告 |
| 用户说「不确定级别」「不知道是P0还是P1」 | 使用快速决策树辅助判断 | 宁高勿低原则 |
| 30分钟无进展 | 自动升级处理，通知更高级别负责人 | [待确认]升级通知方式 |
| 用户说「这不是事故」「误报」 | 记录误报信息，关闭流程 | 保留记录供后续优化告警规则 |

## Inputs
- 告警信息: string, 必填, 告警标题和内容
- 服务名: string, 必填, 发生告警的服务名称
- 时间窗口: string, 必填, 告警触发时间及观察时间段（如"14:00-14:30"）
- 定级结果: string, 可选, P0或P1，默认由决策树判断
- 根因: string, 可选, 定位到的根本原因
- 影响范围: string, 可选, 受损功能、影响用户数、错误率等
- 修复方案: string, 可选, 具体的修复措施
- 复盘报告: string, 可选, 最终输出的复盘文档
