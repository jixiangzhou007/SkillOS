# 工单处理流程

## 核心问题
按照标准流程处理客户工单，确保问题解决、客户满意、信息完整可追溯

## When to use
- keywords: 工单, INC-, 客户投诉, 支持请求, 故障, 问题, ticket, support
- context: 处理客户提交的技术支持工单，需要按照标准流程操作
- excludes: 非客户工单场景（如内部测试、开发任务）、已关闭的工单

## Instructions
Follow these steps in order. Ask the user if anything is marked [待确认].

1. 接收工单后，检查工单号格式是否为 INC- 开头
   - 如果不是 INC- 格式，[待确认：如何处理非标准工单号？]
2. 检查是否存在重复工单
   - 如果发现重复工单，合并到母单，保留所有时间线记录
   - 在母单备注中注明被合并的子单号
3. 判断问题严重程度
   - 如果客户情绪激动：先安抚客户情绪，暂缓深入技术排查
   - 安抚话术示例：“您好，很抱歉给您带来不便，我们非常重视您反馈的问题。我会立即为您跟进，请稍等。”避免在安抚阶段直接进行技术猜测或诊断
   - 如果客户情绪平稳：进入正常排查流程
4. 搜索知识库
   - **必须按产品线和版本号精确过滤搜索结果**，例如：产品线=报表平台，版本号=V5.0
   - 如果找到匹配方案，提供给客户验证
5. 如果知识库无法解决：
   - 收集完整信息：日志、截图、产品版本号
   - 如果信息不全，发送模板邮件索取所需信息
6. 判断是否需要升级二线支持
   - **升级前必须完成情绪安抚和初步信息收集**，除非工单明确标记为紧急故障且客户同意直接升级
   - 如果需要升级：附带完整对话记录、已尝试步骤、环境信息
   - 同步通知客户经理和客户成功团队（严重故障时必须通知）
7. 与 Jira 缺陷单双向关联
   - 在工单中记录关联的 Jira 单号
   - 在 Jira 单中备注对应的 INC- 工单号
8. 关单前必须获得客户确认问题已解决
   - 如果客户未确认，保持工单状态为 open
   - 如果客户确认已解决，关闭工单

## 工单备注规范
- 固定格式：`问题摘要 / 处理步骤 / 结论 / 待办`
- 示例：`报表导出失败 / 已安抚情绪，搜索知识库未找到方案，已收集V5.0日志，待升级二线 / 待确认 / 待办：升级二线`

## Decision routes
| 用户意图/条件 | 执行动作 | 备注 |
|------------|---------|------|
| 客户情绪激动 | 先安抚情绪，暂缓技术排查，使用标准安抚话术 | 避免激化矛盾，不得在安抚时进行技术诊断 |
| 发现重复工单 | 合并到母单，保留所有时间线 | 确保历史可追溯 |
| 信息不全 | 发送模板邮件索取日志、截图、产品版本号 | 标准化信息收集 |
| 知识库搜索 | 按产品线和版本号精确过滤 | 避免返回不相关结果 |
| 升级二线 | 必须附带完整对话、已尝试步骤、环境信息 | 确保二线可直接接手 |
