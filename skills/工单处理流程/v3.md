# 工单处理流程

## 核心问题
按照标准流程处理客户工单，确保问题解决、客户满意、信息完整可追溯

## When to use
- keywords: 工单, INC-, 客户投诉, 支持请求, 故障, 问题, ticket, support
- context: 处理客户提交的技术支持工单，需要按照标准流程操作
- excludes: 非客户工单场景（如内部测试、开发任务）、已关闭的工单

## Instructions
Follow these steps in order. Ask the user if anything is marked [待确认].

1. 接收工单后，检查工单号格式是否为 INC- 开头
   - 如果不是 INC- 格式，[待确认：如何处理非标准工单号？]
2. 检查是否存在重复工单
   - 如果发现重复工单，合并到母单，保留所有时间线记录
   - 在母单备注中注明被合并的子单号
3. 判断问题严重程度
   - 如果客户情绪激动：先安抚客户情绪，暂缓深入技术排查
   - 如果客户情绪平稳：进入正常排查流程
4. 搜索知识库
   - 按产品线和版本号过滤搜索结果
   - 如果找到匹配方案，提供给客户验证
5. 如果知识库无法解决：
   - 收集完整信息：日志、截图、产品版本号
   - 如果信息不全，发送模板邮件索取所需信息
6. 判断是否需要升级二线支持
   - 如果需要升级：附带完整对话记录、已尝试步骤、环境信息
   - 同步通知客户经理和客户成功团队（严重故障时必须通知）
7. 与 Jira 缺陷单双向关联
   - 在工单中记录关联的 Jira 单号
   - 在 Jira 单中备注对应的 INC- 工单号
8. 关单前必须获得客户确认问题已解决
   - 如果客户未确认，保持工单状态为 open
   - 如果客户确认已解决，关闭工单

## Decision routes
| 用户意图/条件 | 执行动作 | 备注 |
|------------|---------|------|
| 客户情绪激动 | 先安抚情绪，暂缓技术排查 | 避免激化矛盾 |
| 发现重复工单 | 合并到母单，保留所有时间线 | 确保历史可追溯 |
| 信息不全 | 发送模板邮件索取日志、截图、产品版本号 | 标准化信息收集 |
| 需要升级二线 | 附带完整对话、已尝试步骤、环境信息 | 同步通知客户经理和客户成功团队 |
| 严重故障 | 同步通知客户经理和客户成功团队 | 必须通知 |
| 客户确认问题已解决 | 关闭工单 | 关单必要条件 |
| 客户未确认 | 保持 open 状态 | 不可强行关单 |

## Inputs
- ticket_id: string, 无默认值，工单号（INC-开头）
- customer_mood: string, 默认值 "calm"，可选值 ["calm", "upset", "angry"]
- severity: string, 默认值 "normal"，可选值 ["low", "normal", "high", "critical"]
- product_line: string, 无默认值，产品线名称
- version: string, 无默认值，产品版本号
- jira_ticket: string, 可选，关联的 Jira 缺陷单号
