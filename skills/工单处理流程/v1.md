# 工单处理流程

## 核心问题
这个流程解决如何从接收客户工单到最终关单的全过程，包括需求转交、SLA 响应、信息补全、升级二线、关单确认和严重故障通知。

## When to use
- keywords: 工单, 故障, 支持, 问题, 报修, 投诉, 需求, 升级, 关单, SLA, P1, P2, P3, INC-
- context: 当用户需要处理客户支持工单、故障报修、功能需求或需要了解工单处理流程时。
- excludes: 当用户询问产品使用方法、一般性咨询、或与工单处理流程无关的对话时，不要使用此技能。

## Instructions
Follow these steps in order. Ask the user if anything is marked [待确认].

1.  接收客户工单或请求。
2.  判断请求类型：
    -   如果是**功能请求**（Feature Request），记录需求详情并转交产品团队处理。**不计入故障 SLA**。
    -   如果是**故障/问题报告**，进入下一步。
3.  判断故障严重等级（P1/P2/P3）：
    -   **P1（严重故障）**：4 小时内首次响应。**必须同步通知客户经理和客户成功团队**。
    -   **P2（一般故障）**：当天首次响应。
    -   **P3（轻微问题）**：三个工作日内首次响应。
4.  检查客户提供的信息是否完整（如日志、截图、产品版本号）：
    -   如果信息不全，使用模板邮件向客户索取缺失信息（日志、截图、产品版本号）。
    -   如果信息完整，进入下一步。
5.  尝试根据现有信息进行初步排查和解决。
6.  如果问题无法解决，需要升级到二线支持：
    -   升级前必须确保附带**完整对话记录、已尝试的解决步骤、环境信息**。
7.  在工单处理过程中，可以创建 Jira 缺陷单进行双向关联。工单号格式为 `INC-` 开头。
8.  在准备关单前，**必须获得客户确认问题已解决**。
    -   如果客户确认已解决，关闭工单。
    -   如果客户未确认或问题未解决，保持工单状态为 `open`，并继续处理。

## Decision routes
| 用户意图/条件 | 执行动作 | 备注 |
|------------|---------|------|
| 用户报告一个功能请求 | 记录需求并转交产品团队 | 不计入故障 SLA |
| 用户报告一个故障，且信息不全 | 发送模板邮件索取日志、截图、产品版本号 | 等待客户回复 |
| 用户报告一个故障，且信息完整 | 进行初步排查和解决 | 若无法解决，准备升级 |
| 用户报告一个故障，需要升级二线 | 附带完整对话、已尝试步骤、环境信息后升级 | 升级前必须准备齐全 |
| 用户报告一个故障，且为 P1 严重故障 | 执行 P1 流程，并通知客户经理和客户成功团队 | 4 小时内首次响应 |
| 用户报告一个故障，且为 P2 一般故障 | 执行 P2 流程 | 当天首次响应 |
| 用户报告一个故障，且为 P3 轻微问题 | 执行 P3 流程 | 三个工作日内首次响应 |
| 用户确认问题已解决 | 关闭工单 | 关单前必须获得确认 |
| 用户未确认问题已解决 | 保持工单 open，继续处理 | 不得强制关单 |

## Inputs
- ticket_type: string, 默认值 "故障", 说明：工单类型，可选值 ["故障", "功能请求"]
- severity: string, 默认值 "P3", 说明：故障严重等级，可选值 ["P1", "P2", "P3"]
