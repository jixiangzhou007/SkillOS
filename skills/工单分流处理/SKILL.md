# 工单分流处理

## 核心问题
根据工单类型（故障、咨询、功能请求）自动执行不同的处理流程，包括知识库匹配、SLA监控、升级转交和关单确认，确保工单高效闭环。

## Instructions
Follow these steps in order. Ask the user if anything is marked [待确认].

1.  **识别工单类型**：读取工单内容，判断属于以下三类之一：
    *   **故障类**：系统报错、服务不可用、功能异常等。
    *   **咨询类**：询问使用方法、配置、最佳实践等。
    *   **功能请求**：要求新增功能、修改现有功能、提出产品建议。

2.  **故障类处理流程**：
    *   **步骤 2.1**：在 Confluence 知识库中搜索与故障描述匹配的已知问题或解决方案。
        *   **步骤 2.1.1**：**信息补全与过滤**：在搜索前，确认工单中是否包含**产品线**和**版本号**。如果信息缺失，按照步骤5“信息补全”执行，并在获取信息后，使用该信息作为过滤条件进行搜索。
    *   **步骤 2.2**：如果找到匹配方案，执行方案并更新工单。
    *   **步骤 2.3**：如果 **30分钟内** 未找到方案或无法解决，将工单升级至二线工程师。
    *   **步骤 2.4**：根据故障严重等级（P1/P2/P3）启动SLA计时：
        *   P1（严重故障）：4小时内首次响应。**同时**，必须同步通知客户经理和客户成功团队。
        *   P2（一般故障）：当天内首次响应。
        *   P3（轻微故障）：三个工作日内首次响应。

3.  **咨询类处理流程**：
    *   **步骤 3.1**：在 FAQ 库中搜索与咨询内容匹配的问答。
    *   **步骤 3.2**：计算匹配度。如果匹配度 **≥80%**，直接回复FAQ内容给用户。
    *   **步骤 3.3**：如果匹配度 **<80%**，将工单转交给人工客服处理。

4.  **功能请求处理流程**：
    *   **步骤 4.1**：记录用户的需求和期望功能。
    *   **步骤 4.2**：将需求转交给产品团队。
    *   **步骤 4.3**：**注意**：功能请求不计入故障SLA。

5.  **信息补全**：
    *   如果工单信息不全（例如缺少日志、截图、**产品线**、**版本号**），使用模板邮件向用户索取。
    *   **工单号格式**：所有工单号应以 `INC-` 开头 [待确认是否强制]。
    *   **与Jira关联**：故障类工单如果确认为缺陷，应创建Jira缺陷单，并在工单和Jira缺陷单之间建立双向关联 [待确认具体实现方式]。

6.  **关单确认**：
    *   在关闭工单前，必须获得客户明确确认问题已解决。如果客户未确认，工单应保持 `open` 状态。
