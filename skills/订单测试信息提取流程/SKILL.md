# 技能名称：订单测试信息提取流程 (Order Test Extraction Flow)

## 触发场景
- **主触发器**: 当用户提交订单时自动触发。
- **辅助触发器**: 支持手动重试（如通过管理后台点击“重新提取”按钮）和定时任务（如每小时扫描未处理的测试订单）。
- [待补充] 手动重试是否需要权限校验？

## 输入/前置条件
- **核心输入**: 用户提交的完整订单数据（通常为JSON或结构化数据格式）。
- **关键字段**: 订单需包含“特殊备注”（字段名：`specialNote`）或类似的、可承载用户测试意图的字段。
- **字段约束**:
  - `specialNote` 字段类型：字符串（String），最大长度不超过500字符。
  - 订单必须包含以下必填字段：`orderId`（订单ID，字符串）、`customerId`（客户ID，字符串）、`status`（订单状态，枚举值：pending/paid/confirmed/cancelled）。
- **前置校验**:
  - 必须确认订单数据已成功提交并持久化（即`orderId`在数据库中已存在）。
  - [待补充] 是否需要验证订单用户权限（如：仅特定角色可发起测试请求）？

## 执行步骤
1.  **输入校验**: 检查订单数据中与“测试意图”相关的字段（如`specialNote`）是否存在且非空。
    - 若字段不存在或为空，流程终止，并记录日志：“未检测到测试意图，跳过提取”。
    - 若字段存在且非空，进入下一步。
2.  **意图解析**: 解析`specialNote`字段内容，识别是否包含明确的测试请求关键词（如“测试”、“验证”、“检查”）。
    - [待补充] 使用简单的关键词匹配还是调用NLP模型进行语义理解？
    - 若未识别到测试意图，流程终止，并标记订单为“非测试订单”。
3.  **信息提取**: 
    - **步骤3a**: 在提取前，校验`specialNote`字段的格式和长度，防止恶意注入或过长文本导致解析失败。
    - **步骤3b**: 同时检查订单的`status`字段，确保订单处于可提取状态（如“已支付”或“已确认”），防止在订单未完成时重复提取。
    - **步骤3c**: 根据预定义或动态识别的测试需求，从订单的其他结构化字段（如`productId`、`price`、`quantity`）中提取相关信息。
    - [待补充] 提取出的信息应如何格式化？是否需要对提取结果进行二次校验（如：提取的价格是否与备注一致）？
4.  **结果输出**: 将提取成功的测试信息封装为标准化事件或数据结构，发送给下游的测试执行系统。
    - [待补充] 输出格式是JSON、消息队列消息，还是直接写入数据库？

## 输出/产出
- **成功输出**: 一个标准的“测试任务”对象，包含：订单ID、提取的测试参数（商品、价格等）、用户原始备注。
- **失败输出**: 返回错误码和错误描述（如“INVALID_NOTE_FORMAT”、“ORDER_STATUS_INVALID”）。

## 边界情况与异常处理
- **特殊备注为空**: 跳过处理，记录日志。
- **特殊备注超长**: 截断至500字符后继续处理，并记录警告日志。
- **订单状态异常**: 返回错误，提示“订单状态不允许提取”。
