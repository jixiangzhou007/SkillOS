# 技能名称：订单提交测试提取流 (Order Submit Test Extraction Flow)

## 触发条件
- 用户提交订单时触发

## 目标
从订单提交事件中提取关键测试信息，为后续测试执行提供结构化数据。

## 步骤

### 步骤1：确认提取来源与模板
- **输入**：用户提交的订单数据（通常为JSON/XML格式）
- **动作**：询问用户是否有参考模板或示例数据，确认字段结构（如：商品ID、数量、金额、用户ID、支付方式等）
- **输出**：确认后的字段映射表

### 步骤2：定义提取规则
- **输入**：字段映射表
- **动作**：根据订单类型（如普通商品、虚拟商品、组合商品）定义提取规则
  - 普通商品：提取商品ID、数量、单价
  - 虚拟商品：提取兑换码、有效期
  - 组合商品：提取子商品列表、组合优惠
- **边界条件**：
  - 如果字段缺失 → 标记为“字段缺失异常”，暂停流程
  - 如果字段值为空或格式错误 → 标记为“数据格式异常”，给出建议修复
- **输出**：提取规则列表（含异常处理策略）

### 步骤3：执行提取与校验
- **输入**：实际订单数据 + 提取规则
- **动作**：
  1. 根据规则提取字段值
  2. 校验字段值是否符合规则（如金额>0，数量为整数）
  3. 如果校验失败，根据异常策略处理
- **边界条件**：
  - 如果订单包含多个商品 → 循环提取每个商品
  - 如果订单包含优惠券/折扣 → 提取优惠信息并计算最终金额
- **输出**：提取结果（结构化数据） + 校验报告

### 步骤4：输出测试触发信号
- **输入**：提取结果
- **动作**：将提取结果格式化为测试系统可识别的触发信号（如事件消息、API请求）
- **输出**：测试触发信号（如 `{ "event": "order_submitted", "data": { ... } }`）

## 异常处理
- 字段缺失：记录缺失字段，暂停流程，等待用户补充
- 数据格式错误：记录错误详情，尝试自动修复（如将字符串“100”转为整数100），若无法修复则报错
- 多商品循环中断：如果某个商品提取失败，不影响其他商品，单独标记异常
