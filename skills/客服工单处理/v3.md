# 客服工单处理

## 核心问题
标准化处理客户支持工单，确保工单从创建到关闭的全流程规范执行，包括分类、升级、合并和客户确认。

## When to use
- keywords: 工单, 支持, 问题, 故障, 报修, 投诉, 求助, INC, 客户
- context: 当用户提交技术支持请求、报告系统故障、或需要客服协助时使用
- excludes: 销售咨询、产品购买、账户注册等非技术支持场景

## Instructions
Follow these steps in order. Ask the user if anything is marked [待确认].

1. 接收客户问题，创建工单
   - 工单号格式：INC- 开头，后接数字编号
   - 记录客户基本信息、问题描述、产品名称和版本号
   - 标记客户情绪状态（正常/激动）

2. 搜索知识库
   - 按产品线过滤知识库内容
   - 按版本号过滤知识库内容
   - 如果找到匹配方案，提供给客户并验证

3. 判断是否为重复工单
   - 如果发现已有相同问题的工单，执行合并操作
   - 将当前工单合并到母单（原始工单）
   - 保留所有时间线记录，不删除任何历史信息
   - 在母单中备注合并的子单信息

4. 处理客户情绪
   - 如果客户情绪激动，先进行安抚
   - 暂缓深入技术排查，优先稳定客户情绪
   - 使用标准安抚话术 [待确认：具体话术内容]

5. 判断是否需要升级
   - 如果问题超出当前处理范围，升级到二线支持
   - 升级时必须附带：完整对话记录、已尝试的解决步骤、环境信息
   - 如果为严重故障，同步通知客户经理和客户成功团队

6. 处理信息不全的情况
   - 如果缺少必要信息，发送模板邮件索取：日志、截图、产品版本号
   - 等待客户回复后再继续处理

7. 关闭工单
   - 在关单前必须获得客户确认问题已解决
   - 如果客户未确认，保持工单状态为 open
   - 记录最终解决方案到知识库 [待确认：是否需要]

## Decision routes
| 用户意图/条件 | 执行动作 | 备注 |
|------------|---------|------|
| 客户报告新问题 | 创建新工单，分配 INC- 编号 | 记录完整信息 |
| 发现与已有工单重复 | 合并到母单，保留所有时间线 | 母单编号需记录 |
| 客户情绪激动 | 先安抚情绪，暂缓技术排查 | 使用标准安抚话术 |
| 问题无法解决 | 升级到二线，附带完整上下文 | 严重故障需通知客户经理和 CS 团队 |
| 信息不全 | 发送模板邮件索取日志、截图、版本号 | 等待回复 |
| 客户确认问题解决 | 关闭工单 | 否则保持 open 状态 |
| 严重故障 | 同步通知客户经理和客户成功团队 | 升级二线同时执行 |

## Inputs
- ticket_id: string, 自动生成, 工单编号（INC-格式）
- customer_name: string, 必填, 客户名称
- product_line: string, 必填, 产品线名称
- product_version: string, 必填, 产品版本号
- issue_description: text, 必填, 问题详细描述
- customer_mood: enum[normal|upset], normal, 客户情绪状态
- is_severe: boolean, false, 是否为严重故障
- parent_ticket_id: string, null, 母单编号（合并时使用）
