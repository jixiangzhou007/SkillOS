# 代码审查

## 核心问题
当收到Pull Request时，如何系统化地审查代码改动，确保逻辑正确、测试充分、反馈清晰。

## When to use
- keywords: PR, pull request, 代码审查, code review, 审查代码, 审查PR, review PR, 合并请求, merge request
- context: 收到PR通知时、用户要求审查代码时、提交代码审查请求时
- excludes: 非代码审查场景（如文档审查、设计评审）、用户仅要求解释代码而非审查

## Instructions
Follow these steps in order. Ask the user if anything is marked [待确认].

1. **理解PR描述与改动意图**
   - 阅读PR标题、描述和关联Issue
   - 对比实际代码改动与描述是否匹配
   - 识别是否有非预期的附带修改（如删除日志、格式调整等）
   - 检查PR是否遵循单一职责原则：一个PR只解决一个问题
   - 若描述与改动不匹配：标记为“描述与改动不匹配”，要求提交者澄清或修正描述
   - 若包含非预期附带修改：标记并询问提交者意图，确认是刻意清理还是误删
   - 若PR包含多个无关主题：要求拆分为独立PR或提供充分理由

2. **检查代码逻辑正确性**
   - 评估改动是否解决了目标问题
   - 检查是否引入了副作用或新bug
   - 特别关注边界条件改动（如 `>=` 改为 `>`）
   - 确认逻辑变更的完整性和一致性

3. **确认测试覆盖**
   - 检查是否有新增或修改的测试
   - 确认测试是否覆盖了改动逻辑、边界条件和异常场景
   - 若改动涉及边界条件：需特别关注测试是否覆盖了边界值
   - 若改动正确但无测试覆盖：要求补充测试，否则不批准合并

4. **给出反馈**
   - 基于以上分析，选择以下反馈类型之一：
     - **批准**：改动正确，测试覆盖充分，无副作用
     - **要求修改**：具体说明问题位置、原因和建议修改方案
     - **提出问题**：描述疑虑，要求提交者解释或提供更多上下文

## Decision routes
| 用户意图/条件 | 执行动作 | 备注 |
|------------|---------|------|
| PR描述与代码改动不一致（如描述只提A功能，但代码修改了B） | 标记为“描述与改动不匹配”，要求提交者澄清或修正描述 | 高风险（risk_logic） |
| 改动涉及边界条件（如 `>=` 改为 `>`） | 特别关注测试是否覆盖了边界值（如 failed_count = 3 的场景） | 中等风险（risk_test） |
| 改动看似正确但无测试覆盖 | 要求补充测试，否则不批准合并 | 中等风险（risk_test） |
| 改动包含非预期的附带修改（如修复bug时删除了看似无关的日志） | 在步骤1中主动标记并询问提交者意图，确认是刻意清理还是误删 | 中等风险（risk_clarity） |
| PR包含多个无关主题（如同时修改功能、重构、日志调整） | 在步骤1中标记，要求提交者拆分为独立PR或提供充分理由 | 中等风险（risk_scope） |
| 所有检查通过，改动正确且测试充分 | 批准合并 | 无风险 |

## Inputs
- pr_url: string, 无默认值, PR的链接地址或PR编号
- review_depth: string, 默认值 "full", 审查深度（full=完整审查, quick=快速审查仅检查关键问题）
- strict_mode: boolean, 默认值 true, 是否严格执行测试覆盖要求
