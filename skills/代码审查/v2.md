# 代码审查

## 核心问题
确保PR代码变更符合描述、逻辑完整、测试充分，并给出明确的审查决策。

## When to use
- keywords: 审查代码, code review, PR审查, 审查PR, 代码审查
- context: 收到PR创建或更新通知时；用户要求审查某个PR时
- excludes: 用户要求写代码或调试时；非PR场景的代码讨论

## Instructions
Follow these steps in order. Ask the user if anything is marked [待确认].

1. **合规性检查**：收到PR通知后，首先检查PR描述是否包含必要字段（改动原因、影响范围、测试方法）。
   - 若描述不完整或与代码改动明显不符，直接进入反馈决策的 `reject_incomplete` 路径。
   - 若描述合规，进入下一步。

2. **看描述**：核对PR标题、描述、关联Issue是否一致；确认改动范围与描述匹配。
   - 若改动范围超出描述，标记为 `风险：描述不足`，要求更新描述或拆分PR。
   - 若一致，进入下一步。

3. **查逻辑**：检查代码变更是否覆盖了所有相关调用点（包括缓存、状态管理、API调用）；识别未在描述中提及的隐式依赖。
   - 若发现未修改的隐式依赖（如缓存、事件监听），标记为 `风险：逻辑不完整`，要求补充修改。
   - 若逻辑完整，进入下一步。

4. **确认测试**：检查是否包含对应场景的测试（单元/集成/截图）；确认测试覆盖了正向路径和至少一条边界/异常路径。
   - 若缺少边界测试或仅提供无关截图，标记为 `风险：测试不足`，要求补充或提供可复现步骤。
   - 若测试充分，进入下一步。

5. **给反馈**：基于前三步结果给出决策。

## Decision routes
| 用户意图/条件 | 执行动作 | 备注 |
|------------|---------|------|
| 描述不完整或与改动不符 | 打回，列出必须补充的描述项（改动原因、影响范围、测试证据） | 路径：reject_incomplete |
| 逻辑有缺陷或遗漏依赖 | 打回，指出逻辑缺陷或遗漏的依赖 | 路径：reject_logic |
| 测试覆盖不足 | 打回，指出测试覆盖不足，要求补充 | 路径：reject_test |
| 所有检查通过 | 通过 | 路径：approve |
| 检查通过但有非阻塞性建议 | 有条件通过，附带建议 | 路径：approve_with_suggestions |
| 紧急修复/热修复 | 允许降低审查门槛，但必须至少完成“查逻辑”步骤，并在评论中说明原因 | 特殊路径 |
| 新人首次PR | 可适当放宽，但需在评论中给出明确的改进指引，并标记 `mentor_review` | 特殊路径 |

## Inputs
- pr_url: string，必填，待审查的PR链接
- is_hotfix: boolean，默认false，是否为紧急修复
- is_newcomer: boolean，默认false，是否为新人首次PR
