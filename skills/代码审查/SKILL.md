# 代码审查

## 核心问题
这个流程解决如何系统化地进行代码审查，确保每次PR审查都覆盖背景理解、逻辑检查、测试验证和反馈给出四个关键环节。

## When to use
- keywords: 代码审查, PR审查, code review, 审查PR, 审查代码
- context: 当用户提交Pull Request要求审查，或询问代码审查流程时
- excludes: 当用户询问代码编写、架构设计、部署发布等非审查场景时不要使用

## Instructions
Follow these steps in order. Ask the user if anything is marked [待确认].

1. 查看PR描述，理解改动背景和目的。如果PR描述缺失或过于简略，则[待确认：如何处理？建议方案：在评论中要求提交者补充描述，暂停审查]
2. 检查代码逻辑，确认是否符合预期设计。如果发现逻辑错误或与预期不符，则在评论中标注具体问题位置和原因
3. 验证测试覆盖情况，检查是否有新增或修改的测试用例。如果测试覆盖不足（如新增功能无对应测试），则[待确认：如何推进？建议方案：要求提交者补充测试用例后再合并]
4. 给出审查建议或修改意见，包括：批准合并、要求修改后重新审查、或拒绝合并并说明理由

## Decision routes
| 用户意图/条件 | 执行动作 | 备注 |
|------------|---------|------|
| 用户提交PR要求审查 | 执行四步审查流程，输出审查结论 | 标准流程 |
| PR描述缺失或过于简略 | [待确认]暂停审查并要求补充描述 | 边界情况 |
| 测试覆盖不足 | [待确认]要求补充测试用例 | 边界情况 |
| 代码逻辑与预期不符 | 在评论中标注问题位置和原因，要求修改 | 常见情况 |

## Inputs
- pr_description: string, 必填, PR的描述内容，用于理解改动背景
- code_changes: string, 必填, 代码变更内容，用于逻辑检查
- test_coverage: string, 可选，默认"无", 测试覆盖情况说明
- review_decision: enum, 可选，默认"待定", 取值: ["批准", "需修改", "拒绝"]

## Outputs
- status: enum — 执行结果（success / pending / rejected / escalated）
- message: string — 给用户或下游系统的摘要说明
- audit_notes: string — 关键操作记录（可选）
