# 门诊分诊

## 核心问题
标准化门诊分诊流程，确保患者根据病情严重程度和类型被正确引导至普通门诊、急诊或留观区域，并处理发热门诊复诊及疫情期间的特殊流程。

## When to use
- keywords: 分诊, 门诊, 发热门诊, 复诊, 抗原, 疫情期间, 挂号, 急诊, 留观
- context: 当用户描述或询问医院门诊分诊的工作流程、步骤或规则时。
- excludes: 当用户询问的是住院流程、手术流程、或非门诊区域的护理流程时。

## Instructions
Follow these steps in order. Ask the user if anything is marked [待确认].

1.  **患者到达分诊台**
    -   [待确认] 分诊前是否必须先挂号。
2.  **分诊评估**
    -   [待确认] 评估的具体内容（如：主要症状、生命体征测量等）。
3.  **判断是否为发热门诊复诊病人**
    -   **如果是**：
        1.  查询该病人上次就诊记录及检验结果。
            -   [待确认] 获取记录的方式（护士查询系统 / 病人自行提供）。
        2.  检查结果是否异常或缺失。
            -   **如果结果异常或缺失**：[待确认] 处理方式（如：重新检查、联系医生等）。
        3.  **如果结果正常且完整**：继续执行下一步。
    -   **如果不是**：直接执行下一步。
4.  **判断是否为疫情期间**
    -   **如果是**：
        1.  要求患者进行抗原检测。
            -   [待确认] 抗原检测在流程中的具体位置（如：分诊评估前、分诊评估后）。
        2.  根据抗原检测结果分流：
            -   **如果结果为阳性**：[待确认] 处理路径（如：转至特定诊室、上报等）。
            -   **如果结果为阴性**：继续执行下一步。
    -   **如果不是**：直接执行下一步。
5.  **确定分诊结果**
    -   根据评估结果，将患者引导至：普通门诊、急诊、留观区或其他指定区域。
    -   [待确认] 分诊结果的具体判定标准。

## Decision routes
| 用户意图/条件 | 执行动作 | 备注 |
| :--- | :--- | :--- |
| 患者为发热门诊复诊病人 | 先查询上次就诊记录和检验结果，再根据结果完整性/异常情况处理，最后进入常规分诊流程。 | 确保复诊病人病情稳定或已明确诊断。 |
| 疫情期间发热病人 | 在分诊流程中增加抗原检测步骤，并根据检测结果（阳性/阴性）决定后续分流路径。 | 遵循疫情防控要求。 |
| 常规分诊病人 | 直接进行分诊评估，根据评估结果引导至普通门诊、急诊或留观。 | 标准流程。 |

## Inputs
- patient_type: string, 默认值 "常规", 说明：患者类型（常规、发热门诊复诊）。
- is_pandemic_period: boolean, 默认值 false, 说明：当前是否为疫情期间。
- antigen_result: string, 默认值 "未检测", 说明：抗原检测结果（阳性、阴性、未检测）。
- previous_record_status: string, 默认值 "正常", 说明：复诊病人上次就诊记录和检验结果的状态（正常、异常、缺失）。

## Outputs
- status: enum — 执行结果（success / pending / rejected / escalated）
- message: string — 给用户或下游系统的摘要说明
- audit_notes: string — 关键操作记录（可选）
