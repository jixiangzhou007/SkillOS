# GitHub Pull

## 核心问题
规范团队 Pull Request 的创建、审查与合并流程，确保代码质量与安全。

## When to use
- keywords: PR, pull request, 代码审查, code review, 审查代码, 合并请求
- context: 用户发起 PR 审查请求、提交 PR 链接、要求对代码变更进行审查时
- excludes: 用户仅询问代码审查流程概念、不涉及具体 PR 时；用户要求自动化合并而非审查时

## Instructions
Follow these steps in order. Ask the user if anything is marked [待确认].

1. **审查 PR 描述完整性**
   - 检查 PR 标题是否遵循 `<类型>(<范围>): <描述>` 格式（如 `feat(auth): 增加OAuth登录`）[待确认：类型与范围定义]
   - 检查 PR 描述是否包含变更动机（为什么改）和变更说明（怎么改）
   - 若描述仅包含一句代码改动说明（如“修复bug”），则要求作者补充变更原因和潜在影响范围
   - 若变更涉及安全、权限或数据持久化但描述中未提及，则要求补充安全影响分析

2. **检查 diff 规模**
   - 若变更行数超过 500 行，建议作者拆分 PR，并标记为“建议改进”

3. **执行安全专项检查**
   - 检查是否存在 SQL 注入风险（如拼接字符串查询）→ 若存在，标记为“严重缺陷”，禁止合并
   - 检查是否存在硬编码密钥、Token 或密码 → 若存在，标记为“严重缺陷”，禁止合并
   - 检查是否存在未校验的用户输入直接用于数据库查询或文件操作 → 若存在，标记为“严重缺陷”，禁止合并

4. **检查代码逻辑与异常处理**
   - 检查是否存在明显的逻辑错误、空指针风险或未处理的边界条件
   - 检查新逻辑是否包含 try-catch 或错误处理分支 → 若缺失关键异常处理，标记为“中等缺陷”
   - 检查是否有单元测试覆盖新增逻辑的核心路径 → 若无测试覆盖，标记为“中等缺陷”
   - 检查边界条件（如 null 值、空集合、极限输入）是否被测试覆盖 → 若缺失，标记为“中等缺陷”

5. **给出审查结论与反馈**
   - 阻塞性问题（安全漏洞、逻辑错误）→ 使用 **Request Changes** 并附上具体代码位置和修复建议
   - 风格/非阻塞性问题（变量命名、注释缺失）→ 使用 **Suggestion** 或行内评论，允许作者自行决定是否采纳
   - 建议改进（diff 过大、测试覆盖不足）→ 使用 **Comment** 并标记为“建议改进”，不阻塞合并但要求作者回复
   - 若同时存在阻塞性问题和风格问题，先处理阻塞性问题，风格问题可放在后续迭代中优化
   - 若审查者不确定某个变更是否合理，应标记为“待讨论”，并 @ 相关领域专家

6. **最终批准前确认**
   - 确认 CI 全部通过（绿色状态）
   - 若有 breaking change，确认 PR 描述中标注了 migration 说明

## Decision routes
| 用户意图/条件 | 执行动作 | 备注 |
|------------|---------|------|
| PR 描述缺少变更动机 | 要求作者补充变更原因和影响范围 | 中等缺陷 |
| 存在安全漏洞（SQL注入/硬编码密钥/未校验输入） | 标记为严重缺陷，Request Changes，禁止合并 | 阻塞性 |
| diff 超过 500 行 | 建议作者拆分 PR | 建议改进 |
| 缺少单元测试覆盖新逻辑 | 标记为中等缺陷，Comment 要求补充 | 非阻塞 |
| 风格问题（命名/注释） | 使用 Suggestion 评论，不阻塞合并 | 非阻塞 |
| CI 未全绿 | 等待 CI 通过后再批准合并 | 前置条件 |
| 存在 breaking change 但无 migration 说明 | 要求补充 migration 说明 | 中等缺陷 |

## Inputs
- pr_url: string, 必填, Pull Request 的 URL 地址
- pr_title: string, 可选, PR 标题（从 PR 页面自动提取）
- pr_description: string, 可选, PR 描述内容（从 PR 页面自动提取）
- diff_lines: number, 可选, 变更行数（从 PR 页面自动计算）
- ci_status: enum("pass","fail","pending"), 可选, CI 运行状态
- has_breaking_change: boolean, 默认 false, 是否包含破坏性变更

## Outputs
- status: enum — 执行结果（success / pending / rejected / escalated）
- message: string — 给用户或下游系统的摘要说明
- audit_notes: string — 关键操作记录（可选）
