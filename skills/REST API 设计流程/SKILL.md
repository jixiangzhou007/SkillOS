# REST API 设计流程

## 核心问题
标准化REST API从需求评审到上线的全流程，确保前后端并行开发、接口契约一致、质量达标后上线

## When to use
- keywords: API设计, RESTful, 接口契约, OpenAPI, Swagger, PRD评审, 联调, Mock, 集成测试, 上线检查
- context: 产品经理在Jira创建API需求单时启动本流程；需要PRD文档和前端页面原型作为输入
- excludes: 纯前端需求无需后端API变更、仅数据库变更不涉及API、第三方API对接（非自研API设计）

## Instructions
Follow these steps in order. Ask the user if anything is marked [待确认].

1. **接收PRD与需求评审**
   - 从Jira工单获取PRD文档和前端页面原型（原型必须包含所有交互状态）
   - 理解业务需求，识别核心实体与数据边界
   - 与前端工程师基于PRD和UI原型图，共同评审并确认接口契约（返回数据结构、字段含义、边界条件）
   - **关键冲突处理**：当PRD与前端展示需求不一致时，采用以下策略：
     - 若字段为业务核心属性（如价格），应新增或调整数据模型
     - 若字段为衍生/计算属性（如优惠金额、预估送达时间），优先考虑在查询时实时计算，而非存储冗余数据
     - 若字段属于不同实体，考虑提供单独的子端点或聚合端点

2. **数据模型与API端点设计**
   - 根据确认后的接口契约，设计数据库模型（表结构、字段类型、关系）
   - 设计RESTful API端点（路径、方法、参数、状态码）
   - **规范要求**：
     - URL命名使用kebab-case（如 `/api/v1/merchants/available`）
     - 返回格式统一为JSON
     - 响应体结构：[待确认] 统一包装格式（如 `{ code, message, data }`），需定义code字段的类型和取值范围

3. **编写OpenAPI文档**
   - 使用Swagger编写或更新OpenAPI规范文档，作为前后端联调依据
   - 每个端点标注请求示例和错误码
   - **文档质量要求**：
     - 每个字段的 `description` 必须包含业务含义和计算规则（如“基于商家营业状态动态计算”）
     - `example` 值必须符合实际返回格式（如ISO 8601时间戳）
     - 对于衍生字段，需在 `description` 中注明“不持久化存储，查询时实时计算”

4. **并行开发与Mock联调**
   - 前端使用Prism（根据OpenAPI自动生成Mock Server）并行开发
   - 后端实现API
   - 联调发现接口问题直接修改OpenAPI文档，然后重新生成Mock
   - 联调通过后，部署至测试环境

5. **集成测试**
   - 在测试环境运行Postman collection + pytest集成测试
   - 覆盖率要求达到90%以上才能上线
   - 全部通过后进入上线流程

6. **上线与检查清单**
   - 执行上线前检查清单：
     - ✅ API文档已更新
     - ✅ 监控告警已配置
     - ✅ 回滚方案已确认
   - 部署上线

## Decision routes
| 用户意图/条件 | 执行动作 | 备注 |
|------------|---------|------|
| 用户说“新需求来了，开始设计API” | 执行步骤1：从Jira工单获取PRD和原型，组织需求评审 | 确保原型包含所有交互状态 |
| 用户说“接口契约有冲突，PRD和前端不一致” | 执行步骤1中的关键冲突处理策略，根据字段性质决定处理方式 | 核心属性改模型，衍生属性实时计算，不同实体拆分端点 |
| 用户说“OpenAPI文档写好了，开始开发” | 执行步骤3-4：用Swagger生成文档，前端用Prism启动Mock并行开发 | 文档需包含请求示例和错误码 |
| 用户说“联调发现问题” | 直接修改OpenAPI文档，重新生成Mock，继续联调 | 不跳过文档更新步骤 |
| 用户说“测试通过了，准备上线” | 执行步骤6：检查清单确认后部署上线 | 覆盖率需≥90% |
| 用户说“这个需求不需要API设计” | 不执行本流程 | 仅适用于需要新增或修改API的需求 |

## Inputs
- jira_ticket_id: string, 无默认值，Jira工单ID，用于获取PRD和跟踪进度
- prd_doc_url: string, 无默认值，PRD文档链接，包含业务需求描述
- prototype_url: string, 无默认值，前端页面原型链接，必须包含所有交互状态
- coverage_threshold: number, 默认值90，集成测试覆盖率要求百分比
- api_version: string, 默认值"v1"，API版本号，用于URL路径
