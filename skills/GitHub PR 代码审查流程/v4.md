# 技能名称：GitHub PR 代码审查流程
## Core Problem
规范后端团队（Python + FastAPI）的 Pull Request 从提交到合并的全流程，确保代码质量、工单关联性及 CI 合规。

## S_route
| Intent | Action | Resource |
| :--- | :--- | :--- |
| 触发场景 | 开发者在 GitHub 提 PR，target 分支是 main | GitHub Repository |
| 前置检查 | 1. 检查 PR 描述是否写清：变更动机、测试方式、是否 breaking change<br>2. 检查 PR 是否关联 Jira 工单号<br>3. 检查 CI (pytest, ruff) 是否通过 | PR 描述, Jira, GitHub Actions |
| 行数规则 | 改动超过 500 行必须拆 PR（重命名与功能修复应分 PR） | [待补充] |
| 特殊审批 | 1. 涉及数据库 migration → 必须双人 approve<br>2. 涉及安全（auth/权限）→ 必须 @security 组 review | [待补充] |
| 边界情况 | 1. CI 红灯不允许 merge<br>2. 允许 draft PR 先讨论，不触发合并检查<br>3. hotfix 可走 fast-track，但 merge 后 24h 内补测试<br>4. 依赖升级必须单独 PR，不和其他功能混提 | [待补充] |
| 合并决策 | [待补充] | [待补充] |
