# 严重故障处理

## 核心问题
规范严重故障的处理流程，确保客户经理和客户成功团队同步收到通知，工单按标准格式管理，升级和合并操作有据可依。同时，需要处理客户情绪和特殊值班场景。

## When to use
- keywords: 严重故障, 故障通知, 工单升级, 重复工单, 客户安抚, 数据安全, 合规, INC-, Jira 关联, 知识库搜索, P1, 值班, 情绪激动
- context: 用户报告严重故障、要求升级二线、发现重复工单、或涉及数据安全/合规问题时使用。包含P1级别故障和客户情绪激动场景。
- excludes: 普通咨询、非故障类工单、已关闭工单的后续讨论

## Instructions
Follow these steps in order. Ask the user if anything is marked [待确认].

1.  **客户情绪评估与安抚（新增）**
    - 判断客户情绪状态。如果客户情绪激动（如指责、威胁投诉、重复提问），**优先执行安抚**，暂缓深入技术排查。
    - 安抚标准话术：[待补充，如“我理解您的情况非常紧急，我们马上启动应急流程，请放心”]
    - 记录客户情绪状态到工单备注。

2.  **确认工单是否为严重故障**（原步骤1，调整顺序）
    - 如果是严重故障（如服务中断、数据丢失、安全事件等），执行步骤 3。
    - 如果不是，按普通工单流程处理。

3.  **故障级别判断与值班处理（新增）**
    - 判断故障级别（P1/P2/P3等）。
    - 如果为 **P1（最高优先级）** 且当前为**非工作时间（如周末、深夜）**，值班工程师可直接电话联系客户，无需等待客户经理确认。联系前需确认值班授权。
    - 如果为P1但工作时间，按正常流程通知客户经理。
    - 通知方式：[待确认，如邮件、即时消息、系统通知]
    - 通知内容：工单编号、故障概述、影响范围、当前状态。

4.  **同步通知客户经理和客户成功团队**（原步骤2，重新编号）
    - 通知方式：[待确认]
    - 通知内容：工单编号、故障概述、影响范围、当前状态、客户情绪状态。

5.  **检查工单号格式**（原步骤3，重新编号）
    - 检查工单号格式是否为 `INC-` 开头。
    - 如果格式正确，继续。
    - 如果格式不正确，修正为 `INC-` 开头。

6.  **检查是否与 Jira 缺陷单关联**（原步骤4，重新编号）
    - 如果有关联，建立双向链接（工单→Jira，Jira→工单）。
    - 如果无关联，标记为待关联。

7.  **搜索知识库**（原步骤5，重新编号）
    - 按产品线和版本号过滤，查找已知解决方案。
    - 如果找到匹配方案，记录到工单备注中，并尝试应用。
    - 如果未找到，标记为待二线处理。

8.  **判断是否需要升级到二线支持**（原步骤6，重新编号）
    - 如果需要升级，执行步骤 9。
    - 如果不需要，继续当前处理。

9.  **升级二线**（原步骤7，重新编号）
    - 升级二线时，必须附带以下信息：
      - 完整对话记录
      - 已尝试的解决步骤
      - 环境信息（操作系统、版本、配置等）
      - **客户情绪状态记录（新增）**

10. **检查是否存在重复工单**（原步骤8，重新编号）
    - 如果存在重复工单，合并到母单（保留所有时间线）。
    - 如果不存在，继续。

11. **按固定格式填写工单备注**（原步骤9，重新编号）
    - 格式：`问题摘要 / 处理步骤 / 结论 / 待办`
    - 问题摘要：简要描述故障现象
    - 处理步骤：已执行的操作列表（包含安抚动作）
    - 结论：当前判断或结果
    - 待办：[待补充，如“等待二线确认数据恢复方案”]
