# 技能名称：电商客服退款处理流程

## 来源
- feasibility://single-shot

## 核心问题
当买家申请退款时，客服需要一套清晰的决策流程，确保退款处理合规、高效，同时规避风险（如恶意退款、金额异常、差评威胁等）。

## S_body

### 第一步：核对订单信息（基础检查）
**做什么**：在ERP系统输入买家提供的订单号，查询订单，核对三件事：
1. 订单号是否存在（不存在则告知买家重新提供）
2. 实付金额与申请退款金额是否一致（不一致则标记为“金额异常”，转主管处理）
3. 付款渠道（支付宝/微信）—— 仅用于后续退款路径确认

**为什么重要**：防止输错订单号导致退错款，或金额不一致引发纠纷。

**知识关联**：这一步与 `_verify_import` 技能的「数据核对」步骤相通——都是先验证输入数据的准确性，再进入后续流程。

### 第二步：按发货状态分路径处理（核心决策）
**做什么**：查看ERP中该订单的“发货状态”，按以下三种情况分别处理：

| 发货状态 | 处理方式 | 为什么重要 |
|---------|---------|-----------|
| **未发货** | 在ERP点击“同意退款”，执行秒退 | 未发货退款无风险，快速处理提升体验 |
| **已发货，未签收** | 联系快递公司发起“拦截包裹”请求 | 拦截成功可避免货物损失；拦截失败转主管处理 |
| **已签收** | 进入第三步（七天无理由检查） | 已签收需确认商品状态，防止恶意退款 |

**知识关联**：这一步与 `dispatch-confirm-test` 技能的「状态判断」步骤相通——都是根据订单状态选择不同分支路径。

### 第三步：已签收订单的七天无理由检查
**做什么**：判断买家申请退款的时间是否在签收后的7天内：
- **7天内**：判断退款类型
  - 退货退款：要求买家提供商品完好照片，确认无误后同意退货
  - 仅退款：转主管审批（因为仅退款风险较高）
- **超过7天**：直接转主管审批

**为什么重要**：七天无理由退货是淘宝标准规则，但需要确认商品完好，防止买家恶意退货（如使用后损坏再退）。

### 第四步：边界情况处理（特殊场景）
**做什么**：在以上流程中，遇到以下情况需特殊处理：
1. **退款金额 > 500元**：无论什么情况，都转主管审批（金额大，风险高）
2. **买家威胁差评**：将聊天记录截图，在飞书群升级给主管处理，但不承诺全额退款
3. **虚拟商品（下载链接类）**：告知买家“虚拟商品发货后不支持退款”，拒绝退款
4. **换货申请**：不属于退款流程，转另一个工单类型处理

**为什么重要**：边界情况是退款流程中最容易出问题的环节，提前定义处理规则可以避免客服自行判断导致的风险。

### 第五步：收尾操作（输出物）
**做什么**：退款完成后，执行以下操作：
1. 在ERP标记退款原因分类（质量问题/不喜欢/发错货）
2. 退款完成后24小时内，通过旺旺发送一条模板消息确认
3. 同步更新CRM备注

**为什么重要**：标记原因有助于后续数据分析（如质量问题频发需改进产品）；发送确认消息可减少买家后续咨询；CRM备注便于其他客服了解历史。

**知识关联**：这一步与 `Export Data from App` 技能的「数据标记」步骤相通——都是将操作结果记录到系统中，形成可追溯的记录。

## S_trigger

**触发条件**：
- 买家在旺旺或订单页申请「仅退款」或「退货退款」

**不应触发的边界**：
- 换货申请（走另一个工单类型）
- 买家仅咨询退款政策（未实际申请）
- 售后维修申请（走售后流程）

## S_params

| 参数 | 说明 | 可选值 |
|------|------|--------|
| 订单状态 | 发货进度 | 未发货 / 已发货未签收 / 已签收 |
| 退款类型 | 买家申请的类型 | 仅退款 / 退货退款 |
| 退款金额 | 申请退款的金额 | 任意数值（>500需主管审批） |
| 签收时间 | 买家签收的日期 | 任意日期（>7天需主管审批） |
| 商品类型 | 是否为虚拟商品 | 实物 / 虚拟（下载链接类） |
| 买家行为 | 是否有威胁差评 | 是 / 否 |
