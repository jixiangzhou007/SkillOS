# 我想做一个电商客服退款处

## 核心问题
标准化处理买家发起的退款/退货请求，在风险可控的前提下自动完成退款，减少人工介入。

## When to use
- keywords: 退款, 退货, 要退钱, 退钱, 售后
- context: 用户在飞书或工单中发起退款/退货请求时
- excludes: 非订单类咨询、纯物流查询、换货请求

## Instructions
Follow these steps in order. Ask the user if anything is marked [待确认].

1.  **核实订单与支付信息，并执行极速退款或转人工**
    -   获取订单号，确认用户身份与订单购买人一致。
    -   检查订单支付状态（已支付/未支付）和发货状态（未发货/已发货/已签收）。
    -   检查订单是否在退款/退货时效内（7天无理由、15天质量问题等）。
    -   **if** 订单未支付或已超时，标记为“高风险”，转人工处理。
    -   **if** 订单已支付且未发货，直接进入“极速退款”通道，无需用户提供证据，跳至步骤3。
    -   **if** 订单已发货但用户未收货，优先发起物流拦截。**if** 拦截成功，按“未发货”处理，跳至步骤3；**if** 拦截失败，告知用户等待签收后处理，或引导用户拒收。

2.  **识别退款类型、判定责任，并计算退款金额与风险检查**
    -   **if** 订单已签收且超过7天无理由，要求用户提供商品完好照片/视频，确认无使用痕迹后，发起“退货退款”。
    -   **if** 订单已签收且在7天内，根据用户申请类型（仅退款/退货退款）处理。
    -   要求用户提供证据（照片/视频），判断责任方（商家/物流/买家）。
    -   **if** 责任模糊或用户情绪激烈，标记为“中风险”，建议安抚并提供补偿方案（如优惠券、部分退款）。
    -   根据退款类型和责任判定，计算退款金额：**仅退款**退还实付金额（含运费，若责任在商家）；**退货退款**退还商品金额，运费按责任方承担。
    -   执行风险检查：**if** 退款金额 > 500元或用户历史有异常退款记录或地址异常，标记为“高风险”，转人工复核，流程结束。

3.  **执行退款与闭环**
    -   在后台系统发起退款/退货申请，按支付渠道处理（微信/支付宝原路退回，银行卡走对公打款）。
    -   退款成功后，同步更新 ERP 系统中的订单状态。
    -   发送短信通知用户退款结果及预计到账时间。
    -   在工单中备注退款原因（如“7天无理由退货”、“商品质量问题”），关闭工单。

## Decision routes
| 用户意图/条件 | 执行动作 | 备注 |
|------------|---------|------|
| 订单未发货，用户申请退款 | 极速退款，无需证据 | 自动执行 |
| 订单已发货/已签收，金额≤500元，无风控 | 按规则执行退款或退货退款 | 自动执行 |
| 退款金额>500元，或疑似欺诈，或订单超时 | 转人工复核 | 标记风险等级 |
| 支付渠道为微信/支付宝 | 原路退回 | 自动匹配 |
| 支付渠道为银行卡 | 对公打款 | 需人工确认账户信息 |

## Inputs
- order_id: string, 必填 — 订单编号
- refund_type: enum, 默认 auto — 退款类型：auto(自动识别), full(仅退款), return(退货退款)
- amount: number, 必填 — 申请退款金额（元）
- risk_level: enum, 默认 low — 风险等级：low(低), medium(中), high(高)

## Outputs
- status: enum — 执行结果（success / pending / rejected / escalated）
- message: string — 给用户或下游系统的摘要说明
- audit_notes: string — 关键操作记录（可选）
