# 技能名称：订单信息测试提取流程

## 触发场景
- 当用户提交订单时触发。
- [待补充] 触发来源（如：API调用、Webhook事件、消息队列消费等）。

## 输入/前置条件
- 需要订单ID和客户ID作为输入。
- [待补充] 需要明确的订单数据结构定义（如JSON Schema）。

## 执行步骤
1.  **接收订单**：捕获用户提交的订单事件。
    - [新增] 对订单类型进行初步分类（如：普通订单、赠品订单、补发订单等）。
2.  **前置校验（新增步骤）**：
    - [新增] 检查订单金额是否为0元且无收件人地址 → 若为赠品订单，标记为“待人工审核”或执行特殊提取规则（如跳过地址校验）。
3.  **信息提取**：
    - 提取标准字段（如：商品名称、数量、单价、总价、收件人地址等）。
    - [待补充] 是否需要提取备注、优惠券、特殊要求等非结构化信息。
    - [待补充] 提取规则：如何处理赠品与主商品的关联关系？如何处理备注中的替换/条件性指令？
4.  **格式校验**：[待补充] 是否需要在提取后进行数据格式或完整性的校验。

## 输出/产出
- [待补充] 提取后的数据结构（例如：JSON格式的订单对象，包含商品列表、赠品关联、特殊标记等字段）。
- [待补充] 输出参数的详细定义（字段名、类型、是否必填）。

## 边界情况
- [新增] **0元无地址订单（如赠品）**：需先判断是否为特殊类型，若是则跳过地址校验或标记为待审核。
- [新增] **含备注替换指令的订单**：需决定是否将备注解析为条件性字段，或仅作为原始文本保留。
- [待补充] 如何处理订单中缺失关键字段的情况？
- [待补充] 如何处理订单备注中包含“替代方案”或“条件性指令”的情况？
