# 技能名称：测试提取流程 (Test Extraction Flow)

## 触发条件
- 用户提交订单时触发（事件来源：[待补充] 是API回调、消息队列还是前端操作？上下文传递方式：[待补充]）

## 输入/前置条件
- [待补充] 订单数据（是否包含支付流水、物流状态？字段范围：[待补充]）
- [待补充] 预期行为规则（如优惠券使用规则、价格计算规则）
- **新增：** 异常差异记录（格式：[待补充] 是表格还是键值对？示例：{差异类型: "金额错误", 差异值: 4.70, 预期: 103.20, 实际: 98.50}）

## 执行步骤
1. **异常捕获**：当发现实际结果与预期不符时，记录具体的差异点。例如，“预期优惠金额为50元，实际优惠金额为30元，差异为20元”。
2. **根因定位**：基于异常差异，判断问题可能出在哪个环节（如：优惠规则匹配、金额计算、叠加逻辑等）。
3. **测试用例提取**：将异常场景转化为标准化的测试用例，包含：[待补充] 用例字段结构（如测试ID、操作步骤、预期结果、实际结果、差异类型标签、根因标签、优先级、复现概率、关联异常ID等）。
4. **优先级标记**：根据异常对用户或业务的影响程度，标记用例的紧急等级（高/中/低）。
5. **[待补充] 输出格式**：提取出的测试用例列表（格式待定，建议为结构化表格或JSON数组）。

## 输出/产出
- [待补充] 提取出的测试用例列表（格式待定，需与步骤3的字段结构一致）
- **新增：** 异常差异记录表（用于追踪每次异常，包含异常ID、订单ID、差异描述、根因、时间戳）

## 边界情况

- [待补充] 当优惠券已过期但仍被使用时的测试方向
- **新增：** 当预期结果与实际结果差异极小（如0.01元）时，是否仍需要提取测试用例？需设定“差异阈值”（默认值：[待补充] 如0.01元）。
- **新增：** 当差异为“舍入误差”时，是否应忽略？需定义“舍入误差”的判断标准（如：差异 < 0.005元 且 为四舍五入导致）。

## 备注
- 本流程旨在从实际业务异常中，系统性地提取出可复用的测试场景，而非一次性修复。
- **新增：** 建议使用“异常捕获→根因定位→用例提取→优先级标记”的流水线步骤。
- **新增：** 优先级标记需结合“差异大小”和“对用户的影响”综合判断（如0.01元差异可能为低优先级，但若累计影响大则需提升）。
- **新增：** 测试用例提取环节的字段结构需根据实际业务场景进一步细化（如是否需要“复现概率”字段？）。
