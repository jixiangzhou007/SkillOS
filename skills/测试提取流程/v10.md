# 技能名称：测试提取流程（Test Extraction Flow）

## 触发条件
- **主触发**：当用户提交订单时自动触发。
  - [待补充] 是否所有订单都触发？还是仅特定类型（如“测试任务”类型）的订单？

## 输入/前置条件
- **输入源**：订单中需包含可提取数据的来源。
  - 类型1：附件（如 PDF 测试报告、截图、日志文件）。
  - 类型2：表单文本（用户在订单备注或特定字段中填写的描述）。
- **前置校验**：
  - 必须至少存在一种输入源（附件或文本），否则流程应中断并报错。
  - [待补充] 是否需要校验附件格式白名单（如仅支持 .pdf, .png, .jpg）？

## 执行步骤（初版）
1. **输入验证**：检查订单中是否存在附件或有效文本。
   - 若无输入源 → 返回错误提示：“缺少测试数据源，请上传附件或填写测试描述”。
2. **提取策略选择**：
   - 若存在附件 → 调用文档解析模块（OCR/PDF解析）进行结构化提取。
   - 若只有文本 → 调用自然语言解析模块进行字段匹配。
   - **[新增] 若同时存在附件和文本**：执行用户选择的优先级策略（见边界情况）。
3. **字段提取**：从数据源中提取预设的关键字段。
   - [待补充] 具体字段列表（如：测试项ID、测试结果、执行耗时、状态）。
4. **结果校验**：检查提取结果是否完整、格式是否符合预期。

## 输出/产出
- **结构化数据**：以 JSON 格式输出提取结果。
  - 示例：`{ “test_item”: “login_test”, “result”: “pass”, “duration_ms”: 1200 }`
- **状态标记**：标记流程是否成功，若部分字段缺失，需标注缺失字段名。
- **[待补充] 产出格式**：根据用户对输出格式的选择（纯文本/JSON/PDF/仅系统更新），进一步细化产出规范。

## 边界情况
- **附件损坏或无法解析**：返回指定错误码，并提示用户重新上传。
- **多附件处理**：[待补充] 是合并提取还是仅处理第一个附件？
- **提取结果为空**：即使流程成功，若未提取到任何字段，应返回空结果并提示“未识别到有效测试数据”。
- **[新增] 附件与文本共存策略**：需要用户明确选择以下一种处理方式：
  - 优先解析附件，附件失败再回退到文本。
  - 优先解析文本，文本字段不足再回退到附件。
