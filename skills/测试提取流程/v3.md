# 技能名称：测试提取流程（Test Extraction Flow）
## 描述
围绕“从特定来源（如订单、文档、消息）中提取结构化数据”这一核心任务，定义所需的测试步骤、输入输出规范及异常处理。

## 触发场景（待完善）
- [✅ 已覆盖] 用户提交订单时触发提取流程。
- [待补充] 其他触发方式（如定时任务、手动触发、API调用）。

## 输入/前置条件（待探索）
- [✅ 已覆盖] 需要订单ID和客户ID作为输入。
- [待补充] 需要明确原始数据的格式（如JSON、XML、纯文本、PDF）。
- [待补充] 需要明确提取规则的定义位置（如代码逻辑、配置文件、AI模型）。
- [待补充] 是否有参考模板（如退款政策文档、订单模板）。
- [待补充] 退款原因（如“商品破损”“不想要了”），用于决定退款金额上限。

## 执行步骤（待探索）
- [✅ 已覆盖] 第一步：验证订单（包括验证订单ID、客户ID、订单状态等）。
- [✅ 已覆盖] 第二步：检查退款政策。
- [✅ 已覆盖] 第三步：计算退款金额。
- [✅ 已覆盖] 第四步：执行退款操作。
- [待补充] 建议使用MetaSkill流水线来串联上述步骤。
- [待补充] 第二步“检查退款政策”需明确：政策来源（文档/配置表/AI）、匹配方式、匹配失败的处理。

## 输出/产出（待探索）
- [待补充] 测试报告格式（如Excel、HTML、JSON）。
- [待补充] 提取结果的正确性、完整性、时效性指标。
- [待补充] 每一步的中间输出（如验证结果、政策匹配结果、计算明细）。

## 边界情况（待探索）
- [待补充] 输入数据为空或格式错误。
- [待补充] 提取字段不存在。
- [待补充] 提取字段存在歧义或冲突（如本例中的“配送地址”被错误填充）。
- [待补充] 客户ID与订单归属客户ID不匹配时的处理策略。
- [待补充] 退款原因不合法或不在预设列表中时的处理策略。
