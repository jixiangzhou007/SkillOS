# 需求发布上线

## 核心问题
标准化需求从开发完成到生产上线的审批与部署流程，确保每次发布经过冒烟测试、QA验收、产品确认，并支持蓝绿部署与快速回滚。

## When to use
- keywords: 上线, 发布, 部署, release, 发版, 冒烟测试, 蓝绿部署, 回滚, staging, 预发布, QA验收, 产品确认
- context: 开发人员在GitLab上针对 `release` 分支创建或更新一个合并请求（Merge Request, MR）时触发。用户描述需求发布上线流程时使用。
- excludes: 用户仅讨论开发阶段、代码审查、本地测试等不涉及发布上线流程的场景；用户仅讨论非生产环境（如dev、test）的部署；用户讨论的是紧急热修复（hotfix）流程（需使用独立的热修复发布流程）。

## Instructions
Follow these steps in order. Ask the user if anything is marked [待确认].

1. **冒烟测试（Staging）**
   - 开发完成后，将代码部署至 `staging` 环境。
   - 执行冒烟测试用例（核心用例，如登录、支付、核心业务流），最多重试3次。
   - **若3次重试后仍有全红失败（P0/P1级阻塞性缺陷），则打回开发，流程终止。**
   - **冒烟测试通过标准**：核心功能无P0/P1级阻塞性缺陷。非功能性缺陷（如轻微性能问题、文案错误）需记录但可不阻塞流程，具体由开发/测试/产品协商决定。
   - 冒烟测试通过后，进入下一步。

2. **创建 Release Branch**
   - 冒烟测试通过后，从主分支（如 `main` 或 `master`）创建 `release` 分支。
   - 分支命名规范：[待确认：如 release/v1.2.3]
   - **创建 `release` 分支时，自动触发GitLab CI流水线，将代码部署至预发布环境。**

3. **预发布验收（QA）**
   - 将 `release` 分支部署至预发布（`pre-release`）环境。
   - 通知 QA 团队进行全量验收测试。
   - **QA验收通过后，需在协作工具（如Jira）中标记“验收通过”，并提交测试报告。**
   - 若QA验收不通过，打回开发修复，重新从步骤1开始。

4. **产品确认**
   - QA 验收通过后，通知产品经理（或需求方）进行最终确认。
   - **产品确认环节需在协作工具（如Jira/飞书）中完成审批，审批通过后触发上线流程。**
   - 若产品确认不通过，打回开发修复，重新从步骤1开始。

5. **生产上线（蓝绿部署）**
   - 产品确认无误后，采用蓝绿部署策略上线生产环境。
   - **先切换10%流量到新版本，观察5分钟，确认无异常后全量切换。**
   - 上线后持续监控30分钟，重点关注：[待确认：核心业务指标/错误日志]。
   - **监控期间若发现问题，立即执行回滚操作。**
   - **回滚操作**：在Jenkins一键执行，5分钟内完成。
   - **上线操作需指定操作人，并记录上线时间、版本号、操作人。**

## Decision routes
| 用户意图/条件 | 执行动作 | 备注 |
|------------|---------|------|
| 冒烟测试3次重试后仍有全红失败 | 打回开发，流程终止 | 需通知开发人员并提供失败详情 |
| QA验收不通过 | 打回开发修复，重新从步骤1开始 | 需记录验收不通过的原因 |
| 产品确认不通过 | 打回开发修复，重新从步骤1开始 | 需记录确认不通过的原因 |
| 上线后监控发现问题 | 立即执行回滚操作（Jenkins一键回滚，5分钟内完成） | 回滚后需通知相关方并记录问题 |
| 上线后监控无异常 | 流程结束，记录上线成功 | 记录上线时间、版本号、操作人 |

## Inputs
- MR链接: string, 必填, GitLab MR的完整URL；需校验是否为标准GitLab地址
- 测试报告: string, 必填, 报告文件路径、链接或关键结论摘要；需明确是否包含冒烟测试结果及重试记录
- Jira工单号: string, 必填, 格式如“PROJ-1234”；需校验工单状态是否为“待上线”或“已批准”
- 分支命名规范: string, 可选, [待确认：如 release/v1.2.3]
- 监控指标: string, 可选, [待确认：核心业务指标/错误日志]
- 操作人: string, 必填, 执行上线操作的人员姓名或ID
- 上线时间: string, 必填, 格式如“2024-01-15 14:30:00 UTC+8”
- 版本号: string, 必填, 格式如“v1.2.3”
