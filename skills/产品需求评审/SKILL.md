# 产品需求评审

## 核心问题
标准化产品需求评审流程，确保PRD文档、原型图、竞品分析经过需求澄清、技术评估、UI评估后，三方达成一致排入迭代，有分歧时由产品经理决策。

## When to use
- keywords: 产品需求评审, PRD评审, 需求评审流程, Jira评审单, 需求澄清, 技术评估, UI评估, 排期
- context: 产品经理在Jira中提交评审单，需要评审PRD文档、原型图、竞品分析，并完成需求澄清、技术评估、UI评估、分歧处理和排期决策
- excludes: 非产品需求评审场景（如代码评审、设计评审、测试用例评审）、没有PRD文档或原型图的简单需求沟通

## Instructions
Follow these steps in order. Ask the user if anything is marked [待确认].

1. **触发评审流程**
   - 条件：产品经理在Jira中提交评审单，状态变更为“待评审”
   - 检查输入文件是否齐全：PRD文档（.md或在线文档链接）、原型图（.png/.jpg或可交互原型链接）、竞品分析（如有，PDF或在线文档；如无则标注“无竞品”）
   - 校验PRD文档内容：必须包含“业务目标”、“功能列表”、“业务规则”章节。若缺失，自动驳回并提示“输入文件不达标，请补充完整”
   - 校验原型图精度：需为高保真原型或关键页面线框图（包含用户交互路径）。若不达标，自动驳回并提示“输入文件不达标，请补充完整”

2. **需求澄清（评审会第一阶段）**
   - 产品经理逐条讲解PRD核心功能与业务目标
   - 确认技术、UI、测试等角色对需求理解一致
   - 检查项：
     - 是否有未定义的边界条件（如数据量、并发、异常处理）
     - 业务规则完整性：例如积分扣减策略、优惠券有效期、退款逻辑等，需在PRD中预定义
     - 输入输出约定：接口返回格式、错误码定义、默认值等
   - 输出：澄清后的需求列表，各方签字确认理解无误
   - 责任人：产品经理（主讲）、全体参会人（确认）
   - 若发现PRD信息缺失，输出“待补充事项清单”，要求产品经理补充后重新发起评审

3. **技术与UI并行评估**
   - 3.1 技术评估
     - 可行性分析：评估技术方案是否可行，识别潜在风险（如性能瓶颈、第三方依赖）
     - 工作量估算：按功能点或故事点估算开发人天
     - 输出：技术评估报告，含风险等级（高/中/低）与备选方案
     - 责任人：技术负责人
     - 评估结果分三档：可行 / 需调研 / 不可行。若为“不可行”，直接驳回至产品经理
   - 3.2 UI/UX评估
     - 交互方案评审：评估交互流程是否合理，是否符合用户习惯
     - 视觉设计工作量：估算设计人天
     - 输出：交互方案确认单，含设计稿或原型链接
     - 责任人：UI/UX设计师

4. **分歧处理（产品经理决策机制）**
   - 当技术、UI或业务方存在分歧时，产品经理作为最终决策者
   - 决策前必须完成的动作：
     - 明确冲突的核心矛盾点（如：用户体验 vs 开发成本 vs 上线时间）
     - 收集各方的客观依据（如：数据量实测、用户调研结论、历史数据）
     - 对照产品目标与迭代优先级（如：该功能是核心体验还是边缘优化）
     - 给出决策理由并同步到评审结果
   - 输出：完整的决策记录，包含分歧点、各方观点摘要、决策理由、影响范围

5. **排期或驳回**
   - 通过：三方评估无异议，或经产品经理决策后，排入迭代
   - 驳回：技术评估结果为“不可行”，或输入文件不达标，直接驳回至产品经理
   - 待补充：需求澄清后，若发现PRD信息缺失，则输出“待补充事项清单”，要求产品经理补充后重新发起评审

6. **超时升级机制**
   - 若评审流程超过3个工作日仍未完成，自动升级到产品总监处理

## Decision routes
| 用户意图/条件 | 执行动作 | 备注 |
|------------|---------|------|
| 输入文件不达标（PRD缺失章节、原型图精度不足、竞品分析缺失且未标注“无竞品”） | 自动驳回，提示“输入文件不达标，请补充完整” | 触发步骤1 |
| 技术评估结果为“不可行” | 直接驳回至产品经理，流程终止 | 触发步骤3.1 |
| 需求澄清后PRD信息缺失 | 输出“待补充事项清单”，要求产品经理补充后重新发起评审 | 触发步骤2 |
| 技术、UI或业务方存在分歧 | 产品经理做最终决策，输出决策记录 | 触发步骤4 |
| 三方评估无异议或经产品经理决策后 | 排入迭代 | 触发步骤5 |
| 评审流程超过3个工作日 | 自动升级到产品总监处理 | 触发步骤6 |

## Inputs
- PRD文档: 文件或链接，无默认值，格式要求：Markdown (.md) 或 在线文档链接，内容必须包含“业务目标”、“功能列表”、“业务规则”章节
- 原型图: 文件或链接，无默认值，格式要求：图片 (.png/.jpg) 或 可交互原型链接，需为高保真原型或关键页面线框图（包含用户交互路径）
- 竞品分析: 文件或链接，默认值“无竞品”，格式要求：PDF或在线文档，如无则必须标注“无竞品”
- 评审结果状态: 枚举值，默认值“待评审”，可选值：通过 / 驳回 / 待补充
- 决策记录: 对象，无默认值，包含字段：分歧点、各方观点摘要、决策理由、影响范围

## Outputs
- status: enum — 执行结果（success / pending / rejected / escalated）
- message: string — 给用户或下游系统的摘要说明
- audit_notes: string — 关键操作记录（可选）
