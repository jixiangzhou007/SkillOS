**季度权限审计**
- 须对照**合规标准**（如 SOC2/等保）给出结论
- 过度授权须**回收/撤销（revoke）**并落实最小权限

## S_body
Follow these steps in order. Ask the user if anything is marked [待确认].

1. **审计准备与证据收集**
   - 明确审计范围：确认被审计系统、网络设备、安全设备、应用系统等资产清单
   - 获取合规标准：收集适用的等保2.0基本要求（如GB/T 22239-2019）及企业内部安全基线配置文档
   - 制定检查表：将标准拆解为可核查的技术与管理条目（如密码策略、日志审计、访问控制、漏洞管理）
   - 若审计范围涉及云服务或SaaS平台，额外收集 AWS、GitHub、Slack 等系统的权限配置文档
   - 收集以下证据材料：访问日志、配置快照、漏洞扫描报告（使用Nmap、OpenSCAP等工具生成）、账号权限清单（包括AWS IAM、GitHub仓库权限、Slack频道权限等）
   - 对每项证据，记录来源、时间戳和获取方式
   - 若证据缺失或不完整，标记为「证据不足」并在报告中说明

2. **权限审计与风险发现**
   - 检查最小权限原则：对比实际权限与岗位职责所需的最小权限
   - 检查离职账号：遍历所有系统，确认离职人员账号是否已禁用或删除
   - 检查过度授权：识别权限超出岗位需求的账号
   - 发现高危漏洞或过度授权时，按以下规则分级：
     - **Critical（紧急）**：24小时内必须整改。包括：核心系统存在可远程利用的高危漏洞、离职账号仍拥有生产环境访问权限、超级管理员权限被非授权人员持有
     - **High（高危）**：7天内必须整改。包括：非核心系统存在高危漏洞、权限超出岗位职责但无恶意行为证据、日志审计缺失超过30天
     - **Medium（中危）**：30天内整改。包括：配置弱化但存在补偿措施、权限分配不规范但未造成实际风险
     - **Low（低危）**：建议项，下个周期整改。包括：轻微偏离基线标准、文档不完整但实际操作合规
   - 风险定级需结合「漏洞可利用性」与「资产重要性」两个维度。例如，一个高危漏洞存在于核心DMZ区的关键应用上，应判定为「Critical」；而同样漏洞存在于内部测试网段的非核心系统上，可判定为「High」
   - 在定级前，必须执行「补偿措施评估」步骤。如果存在有效的补偿措施（如严格的网络隔离、应用层二次认证、运维操作审计等），可以酌情降级（如从「High」降至「Medium」），但必须在报告中明确说明补偿措施的有效性及依赖性风险

3. **报告与整改跟踪**
   - 编制审计报告，包含以下内容：审计范围与标准说明、发现项清单（每项含描述、证据截图/日志、风险等级）、整改建议（具体可操作，如「禁用账号xxx」「配置密码策略为12位以上」）、整改时限（Critical 24h, High 7天, Medium 30天, Low 下周期）、复测计划（整改完成后7个工作日内复测）
   - 建立问题清单，跟踪整改闭环
   - 对Critical问题启动专项督办流程，通知安全负责人和系统负责人
   - 若被审计方对风险定级有异议，启动争议处理流程：由第三方安全专家复核

4. **复核与持续改进**
   - 对已整改项进行复测，确认修复有效性
   - 若复测不通过，重新进入整改流程并升级处理
   - 更新基线标准库，纳入本次审计发现的新型风险点
   - 每季度更新一次基线标准库

5. **异常处理：API超时或系统不可用**
   - 若在证据收集或权限审计过程中，调用外部API（如AWS IAM、GitHub API、Slack API）超时或返回错误，则自动重试3次，每次间隔5秒
   - 若3次重试后仍失败，则记录该系统的证据为「收集失败」，在报告中注明原因及影响范围，并继续处理其他系统
   - 若核心系统（如生产环境）API持续不可用，则标记为「审计阻断项」，通知安全负责人人工介入

## S_route
| 用户意图/条件 | 执行动作 | 备注 |
|------------|---------|------|
| 用户要求执行等保审计 | 执行步骤1-5，以等保2.0标准为检查依据 | 默认使用GB/T 22239-2019 |
| 用户要求执行内部基线检查 | 执行步骤1-5，以企业内部基线文档为检查依据 | 需用户提供基线文档 |
| 用户要求执行权限审计（AWS/GitHub/Slack） | 重点执行步骤2的权限审计部分，输出权限审计专项报告 | 需用户提供各系统的权限导出文件 |
| 发现Critical风险项 | 立即通知安全负责人，启动24小时整改督办流程 | 跳过常规报告流程 |
| 证据缺失或不完整 | 标记为「证据不足」，在报告中说明缺失项及影响 | 不影响其他项的审计 |
| 被审计方对定级有异议 | 启动第三方复核流程，由独立安全专家重新评估 | 复核期间原定级暂不执行 |
| 复测不通过 | 重新进入整改流程，升级至安全总监级别督办 | 记录复测失败原因 |
| API超时或系统不可用 | 自动重试3次，失败后标记为「收集失败」并继续 | 核心系统持续不可用则通知人工介入 |

## S_params

- api_retry_interval: int, 默认5, 每次重试间隔（秒）

- risk_level_high_days: int, 默认7, High级别整改时限（天），与risk_threshold_high含义一致，作为统一参数名

- baseline_update_interval_days: int, 默认90, 基线标准库更新周期（天），与baseline_update_frequency含义一致，作为统一参数名
- api_retry_max_attempts: int, 默认3, API调用失败时的最大重试次数，与api_retry_count含义一致，作为统一参数名
- api_retry_delay_seconds: int, 默认5, 每次重试间隔（秒），与api_retry_interval含义一致，作为统一参数名
- api_timeout_seconds: int, 默认30, 单次API调用超时时间（秒），新增参数
- critical_escalation_contact: string, 默认"security_lead", Critical问题通知对象，可选"security_lead"（安全负责人）、"system_owner"（系统负责人）、"both"
- dispute_reviewer_role: string, 默认"third_party_expert", 争议定级复核人角色，可选"third_party_expert"（第三方专家）、"internal_senior_auditor"（内部高级审计员）
