# 工单分类处理

## 核心问题
当客户通过邮件或Zendesk提交技术支持工单时，如何根据工单类型（故障、咨询、功能请求）自动执行对应的处理流程，并遵守SLA响应时间要求。

## When to use
- keywords: 工单, Zendesk, 技术支持, 故障, 咨询, 功能请求, SLA, 升级, 知识库, FAQ, 二线工程师, 产品团队
- context: 当用户提交或讨论技术支持工单处理流程时使用；当需要根据工单类型自动分流处理时使用
- excludes: 非技术支持类工单（如财务、人事、采购）；已由人工确认处理的工单；仅查询

## Instructions
Follow these steps in order. Ask the user if anything is marked [待确认].

1. **识别工单类型**：读取工单标题和描述，判断属于以下三类之一：
   - 故障（系统不可用、报错、功能异常等）
   - 咨询（如何使用、配置、最佳实践等）
   - 功能请求（新功能建议、需求变更等）
   - 若无法明确分类，标记为「待确认」并转人工审核

2. **信息完整性检查**：在处理任何工单类型前，检查客户是否提供了必要信息（如日志、截图、产品版本号）。
   - 若信息不全，使用模板邮件向客户索取缺失信息，并暂停SLA计时。
   - 若信息完整，继续执行后续步骤。
   - [待确认] 信息不全时，SLA是否暂停或继续计时。

3. **若工单类型为「故障」**：
   - 3.1 查询 Confluence 知识库，搜索与故障描述匹配的已知解决方案
   - 3.2 若在 30 分钟内找到匹配方案，直接回复客户并关闭工单
   - 3.3 若 30 分钟内未找到方案，升级至二线工程师处理
   - 3.4 记录故障工单的 SLA 计时（见步骤 6）

4. **若工单类型为「咨询」**：
   - 4.1 匹配 FAQ 知识库，计算问题与 FAQ 条目的匹配度
   - 4.2 若匹配度 ≥ 80%，直接回复 FAQ 答案并关闭工单
   - 4.3 若匹配度 < 80%，转人工客服处理
   - 4.4 咨询类不计入故障 SLA，[待确认] 咨询类是否有独立 SLA

5. **若工单类型为「功能请求」**：
   - 5.1 记录客户需求详情（功能描述、期望优先级、业务价值）
   - 5.2 转交产品团队进行需求评估
   - 5.3 告知客户已记录需求，[待确认] 是否需要提供预计反馈时间
   - 5.4 功能请求不计入故障 SLA

6. **SLA 响应时间要求（仅适用于故障类工单）**：
   - P1（严重故障）：4 小时内首次响应
   - P2（一般故障）：当天内首次响应
   - P3（轻微故障）：3 个工作日内首次响应
   - [待补充] 如何定义P1/P2/P3的具体标准
