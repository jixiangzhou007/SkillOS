# 工单分类处理

## 核心问题
这个流程解决如何对客户提交的技术支持工单进行自动分类、分派和SLA跟踪，确保不同类型的请求得到正确的处理路径。

## When to use
- keywords: 工单, Zendesk, 技术支持, 故障, 咨询, 功能请求, SLA, 升级, 知识库, FAQ
- context: 当客户通过邮件或 Zendesk 工单系统提交问题，需要自动分类和处理时使用
- excludes: 非技术类客服工单（如账单、账户管理）、已由人工处理的工单、内部测试工单

## Instructions
Follow these steps in order. Ask the user if anything is marked [待确认].

1. 识别工单类型：
   - 读取工单标题和描述内容，判断属于以下三类之一：
     - **故障**：系统报错、服务不可用、功能异常等
     - **咨询**：如何使用、配置说明、最佳实践等
     - **功能请求**：希望新增功能、改进现有功能等
   - 如果无法明确分类，标记为「待人工确认」并转人工客服

2. 故障类处理：
   - 立即在 Confluence 知识库中搜索相关故障解决方案
   - 如果 30 分钟内找到匹配方案 → 回复客户并关闭工单
   - 如果 30 分钟内未找到方案 → 升级至二线工程师团队，并通知客户

3. 咨询类处理：
   - 在 FAQ 库中匹配客户问题
   - 如果匹配度 ≥ 80% → 自动回复 FAQ 答案并关闭工单
   - 如果匹配度 < 80% → 转接人工客服处理

4. 功能请求类处理：
   - 记录客户需求详情（功能描述、使用场景、期望优先级）
   - 转交产品团队进行需求评估
   - 不计入故障 SLA 统计

5. SLA 响应时效跟踪：
   - P1（紧急故障）：4 小时内首次响应
   - P2（普通故障）：当天（24 小时内）首次响应
   - P3（咨询/其他）：3 个工作日内首次响应
   - 超时未响应 → 自动升级通知 [待确认：升级通知的具体接收人]

## Decision routes
| 用户意图/条件 | 执行动作 | 备注 |
|------------|---------|------|
| 工单描述为系统故障/报错 | 查 Confluence 知识库，30分钟无解升级二线 | 记录开始时间用于SLA计算 |
| 工单描述为使用咨询 | 匹配 FAQ，匹配度≥80%自动回复，<80%转人工 | 匹配度阈值固定为80% |
| 工单描述为功能请求 | 记录需求并转产品团队 | 不计入故障SLA |
| 工单类型无法识别 | 标记待确认并转人工客服 | 避免误分类 |
| SLA 超时（P1>4h/P2>24h/P3>3工作日） | 触发升级通知 | [待确认]通知对象和方式 |

## Inputs
- ticket_id: string, 必填，工单唯一编号
- ticket_title: string, 必填，工单标题
- ticket_description: string, 必填，工单详细描述
- ticket_source: string, 默认值 "email"，工单来源（email/zendesk）
- sla_priority: string, 默认值 "P3"，优先级（P1/P2/P3），根据工单内容自动判断
- faq_match_threshold: number, 默认值 80，FAQ匹配度阈值百分比
- escalation_timeout_minutes: number, 默认值 30，故障类知识库搜索超时时间（分钟）
