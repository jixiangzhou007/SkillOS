# 工单分类处理

## 核心问题
根据工单类型自动执行不同的处理流程，确保故障类工单在30分钟内完成知识库查询并决定是否升级，咨询类工单通过FAQ匹配决定是否转人工，功能请求类工单记录需求并转产品团队，同时遵守SLA响应时间和关单前客户确认规则。

## Instructions
Follow these steps in order. Ask the user if anything is marked [待确认].

1. 识别工单类型：
   - 优先级规则：当用户描述同时包含多个类型关键词时，按以下优先级处理：**故障类 > 咨询类 > 功能请求类**。因为故障类通常影响业务，需优先排查。
   - 如果用户描述中包含“故障”、“报错”、“无法使用”、“系统崩溃”等关键词 → 归类为“故障类”
   - 如果用户描述中包含“咨询”、“请问”、“如何”、“怎么操作”等关键词 → 归类为“咨询类”
   - 如果用户描述中包含“建议”、“想要”、“希望增加”、“能不能加”等关键词 → 归类为“功能请求类”
   - 如果无法判断 → 标记为[待确认]，询问用户工单类型

2. 故障类处理流程：
   - 第一步：在Confluence知识库中搜索相关故障解决方案。**搜索时必须按“产品线 + 版本号 + 错误关键词”组合过滤**，例如“财务模块 v3.2.1 生成报表 报错500”。
   - 第二步：如果30分钟内找到方案 → 执行方案并记录
   - 第三步：如果30分钟内未找到方案 → 升级至二线工程师
   - 第四步：如果是严重故障（P1级别）→ 同步通知客户经理和客户成功团队
   - 第五步：记录工单号，格式为 INC- 开头（如 INC-20240101-001）
   - 第六步：如果信息不全 → 使用模板邮件索取日志、截图、产品版本号

3. 咨询类处理流程：
   - 第一步：在FAQ库中匹配用户问题
   - 第二步：如果匹配度 ≥ 80% → 直接回复FAQ答案
   - 第三步：如果匹配度 < 80% → 转人工客服处理
   - 第四步：记录工单号，格式为 INC- 开头

4. 功能请求类处理流程：
   - 第一步：记录用户需求详情（功能描述、期望实现时间、业务价值）
   - 第二步：转交产品团队
   - 第三步：告知用户已记录需求，不计入故障SLA
   - 第四步：记录工单号，格式为 INC- 开头

5. SLA响应时间管理：
   - P1（严重故障）：4小时内首次响应
   - P2（一般故障）：当天内首次响应
   - P3（轻微故障）：3个工作日内首次响应
   - 功能请求类：不计入

6. 工单信息补充与关联（增量）：
   - 当工单信息不全时（如缺少产品版本号、错误截图、操作日志），使用标准模板邮件向客户索取，并注明“请提供产品版本号、错误截图、操作日志”。
   - 工单号格式统一为 **INC-** 开头（如 INC-20240515-003）。如果故障在Jira中创建了缺陷单，需在工单备注中**双向关联**Jira缺陷单号，格式为“关联Jira: [JIRA-1234]”。
   - 严重故障（P1）处理完毕后，**必须同步通知客户经理和客户成功团队**，确保客户侧和内部支持团队信息一致。
