  1. 识别工单类型：根据客户描述，判断属于故障、咨询还是功能请求。首先检查工单号格式是否为“INC-”开头，若非此格式则标记为“格式异常，需人工确认”。若判断为故障类工单，则立即根据故障影响范围与紧急程度确定SLA优先级（P1：系统不可用/数据丢失；P2：核心功能受损；P3：非核心功能异常），并启动对应SLA计时。随后在Confluence知识库中搜索匹配的解决方案（搜索时限为30分钟）；若30分钟内未找到有效方案，则升级至二线工程师处理。若无法明确分类，则标记为“待人工确认”。注意：功能请求类工单不计入故障SLA，不启动故障计时。
  2. 工单合并：检查当前工单标题、描述及客户信息，判断是否为已有工单的重复提交；若为重复工单，则将其合并至母单（原始工单），并更新母单的处理状态。合并完成后，需在母单备注中注明已合并的子单ID列表（格式：合并子单：[子单ID1, 子单ID2, ...]），同时保留子单的原始时间线记录（如创建时间、历史操作日志），确保母单可追溯完整处理过程。
  3. 客户情绪处理：若客户在工单描述或沟通中表现出激动、不满等情绪，先安抚客户情绪（如使用标准话术：“非常理解您的情况，我们会尽快为您处理”），再进入故障处理流程。在情绪未平复前，应暂缓深入技术排查，并在工单备注中记录情绪状态（格式：情绪状态：激动/不满，暂缓技术排查）。
  4. 故障处理：针对识别出的工单类型，执行对应的标准故障处理流程。在进入故障处理前，需先检查步骤3中客户情绪状态：若存在情绪未平复记录，则继续安抚直至情绪稳定后再进行技术排查；若情绪已平复或无情绪问题，则正常执行流程。若工单SLA优先级为P1（严重故障），则在启动处理流程的同时，**必须**立即通知客户经理和客户成功团队，确保客户侧同步知晓并协同跟进。此通知为强制步骤，不可省略。
  5. SLA响应：记录工单创建时间（工单号格式：INC-开头），确保在约定的SLA时间内完成首次响应和处理。
  6. 知识库查询：在处理过程中，主动查询知识库中相似
