# 严重故障协同处理

## 核心问题
确保严重故障工单在技术处理的同时，同步通知相关团队，并按照标准流程管理工单生命周期，避免信息遗漏和客户投诉升级。

## When to use
- keywords: 严重故障, 工单, INC-, 客户经理, 客户成功, 升级, 二线, 安抚, 数据安全, 合规, 重复工单, 合并, 母单, 值班工程师, P1
- context: 当用户描述一个需要多团队协同处理的严重技术故障或日常工单处理中涉及合并、升级、特殊值班安排等场景时。

## Instructions
Follow these steps in order. Ask the user if anything is marked [待确认].

1.  **识别工单类型与优先级**：
    -   **判断是否为严重故障**：如系统崩溃、数据丢失、核心功能不可用、影响大量客户。
        -   **如果是**：立即执行步骤2。
        -   **如果否**：按普通工单流程处理，但需注意合并与升级规则。
    -   **判断优先级 (P1/P2/P3)**：根据影响范围和紧急程度确定。例如，P1（严重故障）可以电话联系客户，即使周末只有值班工程师。

2.  **处理重复工单**：
    -   **检查是否存在相同问题的其他工单**：通过搜索问题摘要、客户名、报错信息等关键词。
    -   **如果发现重复工单**：
        -   **执行合并操作**：将当前工单合并到最早创建的母单中。
        -   **保留所有时间线**：确保合并后，所有工单的历史记录（对话、操作、状态变更）都可追溯。
        -   **更新母单备注**：在母单备注中记录新合并的工单号及客户信息。
    -   **如果无重复工单**：继续下一步。

3.  **同步通知**：同时通知客户经理和客户成功团队。通知内容需包含：工单号、客户名称、问题简述、影响范围、当前处理状态。

4.  **创建/关联工单**：
    -   确认工单号格式是否为 `INC-` 开头。
    -   **如果是**：直接使用该工单号。
    -   **如果否**：创建新的 `INC-` 格式工单。
    -   检查是否存在关联的 Jira 缺陷单。
    -   **如果有**：在工单备注中记录 Jira 单号，实现双向关联。
    -   **如果无**：[待确认] 是否需要创建新的 Jira 缺陷单。

5.  **知识库检索**：根据问题描述，按「产品线」和「版本号」过滤知识库，查找已知解决方案或临时规避措施。

6.  **客户情绪评估**：
    -   **如果客户情绪激动**：先进行安抚（如：表达理解、承诺跟进、告知预计响应时间），暂缓深入技术排查。记录客户情绪状态到工单。
    -   **如果客户情绪平稳**：继续进行技术排查。

7.  **技术排查与升级**：
    -   尝试所有已知的排查步骤。
    -   **如果需要升级到二线支持**：必须附带以下完整信息，缺一不可：
        -   **完整的对话记录**：包含与客户及一线同事的所有沟通。
        -   **已尝试的排查步骤及结果**：详细列出每一步操作和输出。
        -   **环境信息**：如系统版本、浏览器、操作系统、日志片段、网络拓扑等。
    -   **特殊值班情况**：如果当前是周末且只有值班工程师，对于P1（严重故障）工单，值班工程师可以直接电话联系客户进行紧急处理。

8.  **工单备注与记录**：
    -   每次操作后，按以下固定格式更新工单备注：
        -   `问题摘要 / 处理步骤 / 结论 / 待办`
    -   示例：
        -   `问题摘要：用户无法登录系统，报错500。`
        -   `处理步骤：1. 清除浏览器缓存；2. 重置用户密码；3. 检查服务器状态。`
        -   `结论：问题由服务器临时过载导致，已恢复。`
        -   `待办：监控服务器资源使用情况。`

9.  **合规与安全处理**：
    -   **涉及数据安全或合规的问题**：必须走专门的合规/安全通道，禁止向客户口头承诺具体的修复时间。
