# 严重故障协同处理

## 核心问题
确保严重故障工单在技术处理的同时，同步通知相关团队，并按照标准流程管理工单生命周期，避免信息遗漏和客户投诉升级。

## When to use
- keywords: 严重故障, 工单, INC-, 客户经理, 客户成功, 升级, 二线, 安抚, 数据安全, 合规, 重复工单, 合并, 母单
- context: 当用户描述一个需要多团队协同处理的严重技术故障或

## Instructions
Follow these steps in order. Ask the user if anything is marked [待确认].

1.  **识别严重故障**：判断工单是否属于严重故障（如系统崩溃、数据丢失、核心功能不可用、影响大量客户）。
    -   **如果是**：立即执行步骤2。
    -   **如果否**：按普通工单流程处理。
2.  **同步通知**：同时通知客户经理和客户成功团队。通知内容需包含：工单号、客户名称、问题简述、影响范围、当前处理状态。
3.  **创建/关联工单**：
    -   确认工单号格式是否为 `INC-` 开头。
    -   **如果是**：直接使用该工单号。
    -   **如果否**：创建新的 `INC-` 格式工单。
    -   检查是否存在关联的 Jira 缺陷单。
    -   **如果有**：在工单备注中记录 Jira 单号，实现双向关联。
    -   **如果无**：[待确认] 是否需要创建新的 Jira 缺陷单。
4.  **知识库检索**：根据问题描述，按「产品线」和「版本号」过滤知识库，查找已知解决方案或临时规避措施。
5.  **客户情绪评估**：
    -   **如果客户情绪激动**：先进行安抚（如：表达理解、承诺跟进、告知预计响应时间），暂缓深入技术排查。记录客户情绪状态到工单。
    -   **如果客户情绪平稳**：继续进行技术排查。
6.  **技术排查与升级**：
    -   尝试所有已知的排查步骤。
    -   **如果需要升级到二线支持**：必须附带以下信息：
        -   完整的对话记录
        -   已尝试的排查步骤及结果
        -   环境信息（如：系统版本、浏览器、操作系统、日志片段）
7.  **重复工单处理**：
    -   检查是否存在描述相同或高度相似的工单。
    -   **如果是重复工单**：将其合并到母单（最早创建的工单）。合并后，保留所有子单的时间线和操作记录。
8.  **工单备注记录**：在处理过程中，按以下固定格式记录备注：
    -   `问题摘要 / 处理步骤 / 结论 / 待办`
9.  **特殊通道处理**：
    -   判断问题是否涉及数据安全或合规。
    -   **如果是**：立即停止当前沟通，将工单转入专门的安全/合规处理通道。**禁止**向客户口头承诺任何修复时间。
    -   **如果否**：继续常规处理流程。

## Decision routes
| 用户意图/条件 | 执行动作 | 备注 |
|------------|---------|------|
| 用户报告系统崩溃/数据丢失 | 执行步骤2（同步通知），然后执行步骤3、4、5、6 | 严重故障标准流程 |
| 用户情绪激动，抱怨问题 | 执行步骤5（安抚），暂缓步骤6 | 优先处理客户情绪 |
| 用户询问修复时间，但问题涉及数据安全 | 执行步骤9（转入特殊通道），禁止承诺时间 | 合规红线 |
| 用户报告的问题与已有工单相同 | 执行步骤7（合并到母单） | 避免重复工作 |
| 一线无法解决，需要升级 | 执行步骤6（升级二线），确保附带完整信息 | 信息完整性要求 |
