# 我想做一个电商客服退款处

## 核心问题
标准化处理买家发起的退款/退货请求，在风险可控的前提下自动完成退款，减少人工介入。

## When to use
- keywords: 退款, 退货, 要退钱, 退钱, 售后
- context: 用户在飞书或工单中发起退款/退货请求时
- excludes: 非订单类咨询、纯物流查询、换货请求

## S_params

### 参数抽象度评委

#### 当前问题诊断
- **硬编码业务规则**：步骤中直接写死了“7天无理由”、“15天质量问题”、“退款金额 > 500元”等阈值，以及“微信/支付宝原路退回，银行卡走对公打款”的渠道映射。这些规则无法通过参数配置动态调整。
- **参数区暴露不足**：当前 Inputs 仅暴露了 `order_id`、`refund_type`、`amount`、`risk_level` 四个基础字段，未将上述业务规则抽象为可配置参数，导致每次业务规则变更（如无理由退货从7天改为15天）都需要修改技能逻辑本身。
- **复用性受限**：若将该技能复用到不同店铺（不同退货时效、不同风控阈值）或不同支付渠道（如新增“数字人民币”），当前硬编码方式无法支持，必须复制技能并手动修改步骤。

#### 改进目标
将步骤中所有硬编码的业务规则（阈值、时效、渠道映射、风险规则）抽象为 `S_params` 中的可配置参数，使技能逻辑与业务规则解耦。参数区应提供合理的默认值，并允许调用方按需覆盖。

#### 改进后 `S_params` 内容

```yaml
## S_params

### 业务规则配置（可覆盖默认值）

#### 1. 退款/退货时效规则
- `refund_time_limits`:
  - `no_reason_return_days`: 7          # 无理由退货时效（天），默认7天
  - `quality_issue_return_days`: 15     # 质量问题退货时效（天），默认15天
  - `overdue_action`: "escalate"        # 超时处理动作：escalate(转人工) / reject(直接拒绝)

#### 2. 风险控制阈值
- `risk_thresholds`:
  - `auto_refund_max_amount`: 500       # 自动退款最大金额（元），超过此值转人工
  - `high_risk_amount`: 500             # 高风险金额阈值（元），与上一条可独立配置
  - `enable_abnormal_history_check`: true   # 是否启用历史异常退款记录检查
  - `enable_address_anomaly_check`: true    # 是否启用地址异常检查
  - `medium_risk_trigger`: "ambiguous_liability_or_emotional"  # 中风险触发条件：ambiguous_liability(责任模糊) / emotional(情绪激烈) / both

#### 3. 支付渠道映射规则
- `payment_channel_mapping`:
  - `wechat`: "original_return"         # 微信支付：原路退回
  - `alipay`: "original_return"         # 支付宝：原路退回
  - `bank_card`: "manual_transfer"      # 银行卡：对公打款（需人工确认）
  - `digital_currency`: "original_return"  # 数字人民币：原路退回（新增渠道示例）
  - `default`: "manual_transfer"        # 未匹配渠道的默认处理方式

#### 4. 责任方运费承担规则
- `shipping_fee_liability`:
  - `seller_responsible_reasons`: ["quality_issue", "wrong_item", "missing_item"]  # 商家责任原因列表
  - `buyer_responsible_reasons`: ["no_reason_return", "change_of_mind"]            # 买家责任原因列表
  - `default_liability`: "seller"       # 责任模糊时默认承担方：seller / buyer / shared

#### 5. 证据要求规则
- `evidence_requirements`:
  - `require_photos_for_return`: true   # 退货退款是否需要照片/视频证据
  - `photo_required_after_days`: 7      # 超过此天数（默认7天）必须提供照片
  - `video_required_for_quality_issue`: true  # 质量问题是否需要视频证据

#### 6. 补偿方案配置
- `compensation_options`:
  - `medium_risk_coupon_amount`: 10     # 中风险场景建议优惠券金额（元）
  - `partial_refund_min_percent`: 30    # 部分退款最低比例（%）
  - `partial_refund_max_percent`: 70    # 部分退款最高比例（%）

### 参数使用说明
- 以上所有参数均提供默认值，调用方可根据业务场景（如不同店铺、不同商品类目）在调用时覆盖。
- 参数变更无需修改技能步骤逻辑，仅需调整 `S_params` 中的配置值。
- 若调用方未传入某参数，则自动使用默认值。
```

## Instructions
Follow these steps in order. Ask the user if anything is marked [待确认].

1.  **核实订单与支付信息，并执行极速退款或转人工**
    -   获取订单号，确认用户身份与订单购买人一致。
    -   检查订单支付状态（已支付/未支付）和发货状态（未发货/已发货/已签收）。
    -   检查订单是否在退款/退货时效内（7天无理由、15天质量问题等）。
    -   **if** 订单未支付或已超时，标记为“高风险”，转人工处理。
    -   **if** 订单已支付且未发货，直接进入“极速退款”通道，无需用户提供证据，跳至步骤3。
    -   **if** 订单已发货但用户未收货，优先发起物流拦截。**if** 拦截成功，按“未发货”处理，跳至步骤3；**if** 拦截失败，告知用户等待签收后处理，或引导用户拒收。

2.  **识别退款类型、判定责任，并计算退款金额与风险检查**
    -   **if** 订单已签收且超过7天无理由，要求用户提供商品完好照片/视频，确认无使用痕迹后，发起“退货退款”。
    -   **if** 订单已签收且在7天内，根据用户申请类型（仅退款/退货退款）处理。
    -   要求用户提供证据（照片/视频），判断责任方（商家/物流/买家）。
    -   **if** 责任模糊或用户情绪激烈，标记为“中风险”，建议安抚并提供补偿方案（如优惠券、部分退款）。
    -   根据退款类型和责任判定，计算退款金额：**仅退款**退还实付金额（含运费，若责任在商家）；**退货退款**退还商品金额，运费按责任方承担。
    -   执行风险检查：**if** 退款金额 > 500元或用户历史有异常退款记录或地址异常，标记为“高风险”，转人工复核，流程结束。

3.  **执行退款与闭环**
    -   在后台系统发起退款/退货申请，按支付渠道处理（微信/支付宝原路退回，银行卡走对公打款）。
    -   退款成功后，同步更新 ERP 系统中的订单状态。
    -   发送短信通知用户退款结果及预计到账时间。
    -   在工单中备注退款原因（如“7天无理由退货”、“商品质量问题”），关闭工单。

## Decision routes
| 用户意图/条件 | 执行动作 | 备注 |
|------------|---------|------|
| 订单未发货，用户申请退款 | 极速退款，无需证据 | 自动执行 |
| 订单已发货/已签收，金额≤500元，无风控 | 按规则执行退款或退货退款 | 自动执行 |
| 退款金额>500元，或疑似欺诈，或订单超时 | 转人工复核 | 标记风险等级 |
| 支付渠道为微信/支付宝 | 原路退回 | 自动匹配 |
| 支付渠道为银行卡 | 对公打款 | 需人工确认账户信息 |

## Inputs
- order_id: string, 必填 — 订单编号
- refund_type: enum, 默认 auto — 退款类型：auto(自动识别), full(仅退款), return(退货退款)
- amount: number, 必填 — 申请退款金额（元）
- risk_level: enum, 默认 low — 风险等级：low(低), medium(中), high(高)

## Outputs
- status: enum — 执行结果（success / pending / rejected / escalated）
- message: string — 给用户或下游系统的摘要说明
- audit_notes: string — 关键操作记录（可选）
