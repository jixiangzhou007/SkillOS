# 电商客服退款处理

## 核心问题
当客户发起退款请求时，如何自动完成身份核实、订单状态判断、退款方案匹配与执行，并在超限或异常时转人工处理。

## When to use
- keywords: 退款, 退货, 要退钱, 不想要了, 质量问题, 发错了, 少发了, 坏了, 瑕疵, 破损
- context: 客户在飞书群聊、工单或私聊中发起退款请求时使用
- excludes: 客户仅咨询退款政策但未明确要求退款；客户要求换货但未提及退款；消息内容与退款无关

## 应答速查（单条回复、可执行）

收到「处理客户退款」类请求时，**在同一条回复中**给出完整可执行方案，不要只追问、也不要跳过核实就写「已发起退款」。

**硬性规则**
- 必须先核实订单号、金额、售后期与身份，再进入执行步骤。
- **禁止**在未完成核实时写「已为您发起退款」「直接为您处理退款」。
- 金额 >500 元或疑似欺诈 → 转人工，不自动退款。

**标准应答模板**（按 Input 中的 order / customer 替换占位符）：

```
您好{customer.name}，已收到订单 {order.id}（金额 {order.amount} 元）的退款请求。按退款政策与售后期，执行如下：

1. **核实**：查询订单发货/签收状态与售后期；请客户确认收件人姓名或地址后几位，完成身份校验。
2. **方案**：未发货→仅退款；已发货可拦截→拦截后仅退款；已签收→按质量问题/物流/主观原因匹配退货退款或换货，并收集凭证。
3. **执行**：核实通过后原路退回（微信/支付宝自动原路；银行卡对公打款需人工确认）。
4. **通知**：退款完成后短信通知客户，并同步 ERP 订单状态。

当前状态：待核实。核实通过后进入步骤 3，未完成核实前不得标记为已退款。
```

## Instructions
Follow these steps in order. Ask the user if anything is marked [待确认].

1. **触发与意图识别**
   - 监听飞书群聊、工单或私聊消息。
   - 若消息包含以下关键词之一：退款、退货、要退钱、不想要了、质量问题、发错了、少发了、坏了、瑕疵、破损，则进入退款流程。
   - 若消息同时包含“仅退款”、“退货退款”、“换货”等具体指令词，优先按指令词归类意图。[待确认：是否增加具体指令词作为独立触发条件]

2. **阶段一：身份与订单核实**
   - 提取客户消息中的订单号、手机号或用户名。
   - 调用订单系统API查询订单状态（是否已发货、是否在售后期内）。
   - 与客户确认订单的收件人姓名或地址的后几位，完成身份校验。
   - **决策分支**：
     - 若订单不存在或已超出售后期 → 拒绝请求并告知原因。
     - 若身份校验失败 → 标记为疑似欺诈，转人工处理。

3. **阶段二：退款原因与凭证收集**
   - 根据客户原始描述，自动归类为以下类型之一：
     - **质量问题**（如坏了、不响、瑕疵）
     - **物流问题**（如破损、少发、错发）
     - **主观原因**（如不想要了、买错了、不喜欢）
   - 根据原因类型，自动回复引导话术并要求提供凭证：
     - 质量问题：要求提供故障部位的照片或短视频。
     - 物流问题：要求提供快递面单和外包装破损照片。
     - 主观原因：确认是否影响二次销售，告知运费承担规则。
   - **决策分支**：
     - 若客户拒绝提供凭证 → 提示无法受理，转人工复核。
     - 若凭证明显造假或与描述不符 → 标记为异常。

4. **阶段三：退款方案匹配与执行**
   - 根据订单状态和退款原因，匹配预设方案：
     - **未发货** → 直接走极速退款（仅退款）。
     - **已发货但未签收** → 确认是否拒收或拦截物流；若拦截成功则仅退款，否则进入退货退款流程。
     - **已签收且≤7天无理由** → 按原因类型匹配方案：
       - 质量问题/物流问题 → 退货退款或换货/补发。
       - 主观原因 → 退货退款（客户承担运费）。
     - **已签收且>7天无理由** → 检查商品是否完好，完好则按主观原因处理；否则拒绝。
   - **金额与风险校验**：
     - 若退款金额 > 500元 → 转人工复核，不能自动通过。
     - 若订单被标记为疑似欺诈或异常 → 转人工复核。
   - **支付渠道匹配**：
     - 微信支付 → 原路退回。
     - 支付宝支付 → 原路退回。
     - 银行卡支付 → 对公打款（需人工确认账户信息）。
   - **执行退款**：调用支付网关API执行退款操作。
   - **同步与通知**：
     - 同步更新ERP系统（订单状态、退款原因）。
     - 发送短信通知客户退款结果。

## Decision routes
| 用户意图/条件 | 执行动作 | 备注 |
|------------|---------|------|
| 订单未发货 | 极速退款（仅退款） | 无需凭证 |
| 订单已发货，客户拒收或物流可拦截 | 拦截物流后仅退款 | 需确认物流状态 |
| 订单已签收，质量问题/物流问题 | 退货退款或换货/补发 | 需提供凭证 |
| 订单已签收，主观原因（≤7天无理由） | 退货退款（客户承担运费） | 需确认商品完好 |
| 订单已签收，主观原因（>7天无理由） | 检查商品完好后处理，否则拒绝 | 需人工复核 |
| 退款金额 > 500元 | 转人工复核 | 不能自动通过 |
| 疑似欺诈或异常标记 | 转人工复核 | 需人工判断 |
| 支付渠道为微信/支付宝 | 原路退回 | 自动执行 |
| 支付渠道为银行卡 | 对公打款 | 需人工确认账户信息 |

## Inputs
- order_id: string, 必填，客户提供的订单号
- refund_reason: string, 必填，客户描述的退款原因
- refund_amount: number, 必填，退款金额（单位：元）
- payment_channel: string, 可选，支付渠道（微信/支付宝/银行卡），从订单系统获取
- evidence_urls: array of string, 可选，客户提供的凭证链接
- is_fraud_flagged: boolean, 默认false，是否被标记为疑似欺诈

## Outputs
- status: enum — 执行结果（success / pending / rejected / escalated）
- message: string — 给用户或下游系统的摘要说明
- audit_notes: string — 关键操作记录（可选）
