# 电商客服退款处理

## 核心问题
标准化处理电商平台买家的退款申请，确保退款流程合规、高效，并自动识别需人工复核的异常场景。

## When to use
- keywords: 退款, 退货, 要退钱, 退钱, 申请退款, 退货退款
- context: 用户在飞书或工单中提出退款相关请求时使用
- excludes: 用户咨询商品信息、物流进度、投诉非退款问题时不使用

## Instructions
Follow these steps in order. Ask the user if anything is marked [待确认].

1. **核实订单信息**：
   - 获取订单号，调用订单系统API查询订单状态（未发货/已发货/已签收）、支付方式（微信/支付宝/银行卡/余额）、实付金额及用户身份。
   - 若订单未发货，直接进入步骤3计算全额退款，执行极速退款。
   - 若订单已发货，先确认买家是否已签收。若未签收，优先建议拦截快递；若已签收，进入步骤2。

2. **判定退款原因与类型**：
   - 读取买家填写的退款原因，匹配预设规则：
     - “七天无理由”：无需举证，直接进入步骤3。
     - “商品破损/发错货”：需买家上传开箱照片或视频凭证。
     - “不想要/物流问题”：按订单状态处理。
   - 区分“仅退款”（未发货或已拦截）与“退货退款”（已签收）。
   - **证据冲突处理**：若物流签收记录显示“无异常”但买家主张破损，自动触发举证流程。引导客服要求买家在48小时内上传开箱照片/视频。若超时未举证，默认以物流记录为准，驳回退款；若举证有效，进入人工复核或直接执行退款。

3. **计算退款金额**：
   - 全额退款：未发货订单，直接退实付金额。
   - 部分退款：按商品实付占比计算，公式：商品实付占比 = (商品原价 / 订单总原价) * 订单实付金额。需扣除已使用的优惠券分摊金额。
   - 退款金额上限：不超过实付金额。
   - 若退款金额超过500元，或系统标记为疑似欺诈订单，转人工复核，不能自动通过。

4. **执行退款**：
   - 根据支付渠道执行退款：
     - 微信支付：原路退回。
     - 支付宝：原路退回。
     - 银行卡：走对公打款流程。
   - 若金额超限、状态异常（如已发货未拦截成功）或触发风控，转人工处理。

5. **退款后处理**：
   - 同步更新ERP库存及订单状态为“已退款”。
   - 发送短信通知买家退款结果。
   - 在工单中备注退款原因及处理结果。

## Decision routes
| 用户意图/条件 | 执行动作 | 备注 |
|------------|---------|------|
| 买家发起退款（未发货） | 1. 核实订单信息<br>2. 计算全额退款<br>3. 执行极速退款 | 无需举证，直接秒退 |
| 买家发起退款（已发货/已签收） | 1. 核实订单信息<br>2. 判定退款原因（需举证则触发举证流程）<br>3. 计算退款金额<br>4. 执行退款或转人工 | 需确认物流状态，处理证据冲突 |
| 退款金额>500元或疑似欺诈 | 转人工复核 | 不自动通过，需人工审核 |
| 支付渠道为微信/支付宝 | 原路退回 | 自动执行 |
| 支付渠道为银行卡 | 走对公打款 | 需额外处理 |

## Inputs
- order_id: string, 必填, 订单号
- refund_reason: string, 必填, 退款原因（如“七天无理由”“商品破损”）
- payment_channel: string, 必填, 支付渠道（微信/支付宝/银行卡/余额）
- user_id: string, 必填, 用户ID
- refund_amount: number, 自动计算, 退款金额
- evidence_urls: array, 可选, 买家上传的凭证链接
- result: string, 输出, 处理结果（成功/转人工/驳回）
- notification_content: string, 输出, 发送给买家的短信内容
