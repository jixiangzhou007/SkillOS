# 技能名称：test extraction flow 设计诊断

## 触发场景
- 用户提交订单时触发
- 需要从订单数据中提取特定测试用例字段
- **[待补充] 触发方式是同步还是异步？**

## 输入/前置条件
- [待补充] 订单数据结构定义（字段名、类型、可选性）
- [待补充] 提取规则映射表（源字段 → 目标字段）
- 订单状态枚举值列表（如：pending, confirmed, cancelled, completed）
- **[待补充] 退款政策ID / 政策规则列表**
- **[待补充] 退款金额计算规则（如：全额退款、按比例、有上限）**

## 执行步骤
1. 接收订单数据（JSON/对象格式）
2. **验证订单（当前为最薄弱环节）**
   - **[待补充] 验证规则列表（如：订单是否存在、金额是否合法、状态是否可退款）**
   - **[待补充] 验证失败时的处理策略（如：终止流程、记录日志、返回错误码）**
3. **检查退款政策（当前为最薄弱环节）**
   - **[待补充] 政策匹配逻辑（如：按商品类别、按用户等级）**
   - **[待补充] 政策检查结果（如：允许退款、部分退款、拒绝退款）**
4. 计算金额
   - **[待补充] 计算方式（如：原价 - 已使用积分 - 折扣）**
5. 退款
   - **[待补充] 退款渠道（如：原路返回、余额）**
6. 发邮件通知
   - **[待补充] 邮件模板（成功/失败）**
   - **[待补充] 通知优先级（如：即时、延迟）**

## 输出/产出
- [待补充] 输出格式（JSON Schema 或 CSV 表头）
- [待补充] 输出字段列表

## 边界情况
- [待补充] 字段缺失时的默认值/空值处理
- [待补充] 字段值格式异常（如日期格式不统一）的处理
- [待补充] 同一字段在不同订单中映射不一致的检测机制
- [待补充] 字段值为 null 或空字符串时的处理策略
- 订单状态为“已取消”或“异常”时的处理策略
  - 当前需确认：是否应作为前置校验条件，在步骤1前拦截，或作为特殊分支继续执行
